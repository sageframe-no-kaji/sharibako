# Ho-06.7-AT-02 — CLI: sharibako delete

**Goal**

Add a `sharibako delete <id>` command that deletes a scope, or — with `--shared`
— a shared entry, through the AT-01 Core verbs. Confirmation-gated like `clean`
(mandatory prompt unless `--yes`, TTY-required), no Touch ID. `--force` orphans a
linked shared entry; the un-forced linked case refuses and names the linkers.

**Context**

ho-06.7 Decisions 2, 4, 6 (`ho-process/hos/ho-06.7-delete-verb.md`). Deletion
needs no age key: build `VaultCore` through `init(vaultURL:)`, exactly as
`LinkCommand` does (`Sources/SharibakoCLI/Commands/LinkCommand.swift` — the
keyless, no-Touch-ID precedent, including the `do { try _run() } catch {
ErrorReporter.report(...) }` shape and the `_run()` testable entry point).
Confirmation follows `CleanCommand`
(`Sources/SharibakoCLI/Commands/CleanCommand.swift`): a prompt unless `--yes`, a
`CLIError.promptRequiresTTY` when stdin is not interactive and `--yes` is absent,
`CLIError.aborted` (exit 130) on decline. AT-01 provides
`VaultError.sharedEntryLinked(id:linkers:)`.

**Files**

- Create: `Sources/SharibakoCLI/Commands/DeleteCommand.swift`
- Modify: `Sources/SharibakoCLI/SharibakoCommand.swift` — register `DeleteCommand`
  in the root `subcommands` list.
- Modify: `Sources/SharibakoCLI/Support/ErrorReporter.swift` and/or the exit-code
  type — map `VaultError.sharedEntryLinked` to a message that lists the linkers
  and the "unlink them first, or pass --force to orphan them" hint, and to an exit
  code (reuse the user-error code unless a distinct one reads clearer).
- Modify: the CLI test target (`Tests/SharibakoCLITests/`) — `DeleteCommand` tests
  over temp vaults.

**Required Changes**

1. **`struct DeleteCommand: AsyncParsableCommand`** in `DeleteCommand.swift`.
   - `static let configuration` with `commandName: "delete"`, an abstract, and a
     `discussion` block in the house style (what it removes, that markers and
     materialized `.env` files are left in place, that `sync` commits the removal,
     the `--force` orphan semantics, EXAMPLES, EXIT CODES) — match `LinkCommand`'s
     discussion depth.
   - `@OptionGroup var global: GlobalOptions`
   - `@Argument var id: String` — scope id, or shared-entry id with `--shared`.
   - `@Flag(name: .long) var shared: Bool` — target the shared pool.
   - `@Flag(name: .long) var yes: Bool` — skip the confirmation prompt.
   - `@Flag(name: .long) var force: Bool` — orphan a linked shared entry.
   - `func run() async throws { do { try _run() } catch { ErrorReporter.report(error, json: global.json) } }`
   - `func _run() throws` holding the body (testable-entry convention, matching
     `LinkCommand`).

2. **Body (`_run`)**:
   - Resolve the vault via `VaultLocator.resolve(globalFlag: global.vaultURL)`;
     build `VaultCore(vaultURL:)` — **no `AgeKeyProvider`, no handle, no Touch ID.**
   - Confirmation (skipped when `--yes`): if not `yes`, require an interactive
     stdin — else throw `CLIError.promptRequiresTTY(command: "delete", flag: "--yes")`
     — then prompt naming the target and the blast radius ("Delete scope `<id>`
     and all its secrets? Materialized `.env` files and markers are left in place.");
     `CLIError.aborted` on a no. For `--shared`, phrase for the shared entry.
   - `--shared` → `vault.deleteSharedEntry(id, force: force)`; otherwise
     `vault.deleteScope(id)`. `--force` without `--shared` is a no-op flag on the
     scope path (document it as shared-only; do not error).
   - Success output: human `Deleted scope <id>` / `Deleted shared entry <id>`,
     followed by a line that `sync` will commit the removal; `--json`
     `{"deleted": {"scope": "<id>"}}` / `{"deleted": {"shared": "<id>"}}` via the
     `OutputRenderer`.

3. **Register** `DeleteCommand.self` in `SharibakoCommand`'s `subcommands`
   (placement near the other mutating verbs).

4. **`sharedEntryLinked` reporting** in `ErrorReporter` / exit code: print the id,
   the linkers (`scope/key` per line), and the unlink-or-`--force` hint; map to an
   exit code. Reuse the existing user-error code unless a distinct one is clearer.

5. **Tests** over temp vaults (seed with the Core APIs or fixtures; `--shared`
   tests that need a linked entry create a scope, a shared entry, and a link):
   - `delete <scope> --yes` removes the scope (assert via `listScopes`).
   - `delete <absent> --yes` exits non-zero with `scopeNotFound`'s message.
   - `delete --shared <id> --yes` removes an unlinked shared entry.
   - `delete --shared <linked> --yes` (no `--force`) exits non-zero, message names
     the linker, entry still present.
   - `delete --shared <linked> --yes --force` removes it; the linker `.link` file
     remains (now dangling).
   - Non-TTY without `--yes` throws `promptRequiresTTY` (assert the CLI does not
     delete).

**Acceptance**

- [ ] `sharibako delete <scope>` and `sharibako delete --shared <id>` delete
      through the AT-01 Core verbs, built on `VaultCore(vaultURL:)` — no age key,
      no Touch ID.
- [ ] Confirmation is mandatory unless `--yes`; non-TTY without `--yes` errors
      (`promptRequiresTTY`) and deletes nothing; decline exits 130.
- [ ] Linked shared delete without `--force` refuses, names the linkers, and
      leaves the entry; with `--force` it removes the entry and leaves the linker
      dangling.
- [ ] Success output (human + `--json`) names what was deleted and that `sync`
      commits it.
- [ ] `DeleteCommand` is registered; `sharibako delete --help` renders.
- [ ] Full rhythm green; coverage ≥90%.
- [ ] Zero files under `Sources/SharibakoCore/` or `Sources/Sharibako/` changed.

- Do not auto-commit or run any git — `sync` commits (Decision 3).
- Do not touch markers or materialized `.env` files.
- Do not make `--force` affect scope deletion — it orphans linked *shared*
  entries only.
- Do not add the Core verbs or GUI code — AT-01 and AT-03.

contract needs revisiting, which is a practitioner call.

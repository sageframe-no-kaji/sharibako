# Ho-00-AT-01 — GitHub Actions CI workflow

## Goal

Land a single GitHub Actions workflow at `.github/workflows/ci.yml` that runs the verification stack (swift-format lint, swiftlint, swift build, swift test) on every push to `main` and on every pull request. CI is green on the existing scaffold; nothing in the repo's existing state should fail under this workflow.

## Context

This is a standalone child of ho-00 (orientation). ho-00 is orientation-shaped and doesn't decompose into agent tasks of its own; this task is the one operational followup ho-00 commits to per the Sharibako ho-overview's ho-00 entry. The scaffold already passes all verification commands locally — the job here is to encode that in CI, not to fix anything substantive.

Read before starting:
- `ho-process/hos/ho-00-orientation.md` (parent ho — see "Operational followup")
- `~/.claude/modules/languages-swift.md` (verification rhythm: swift-format lint, swiftlint --strict, swift build, swift test)
- `.pre-commit-config.yaml` at repo root (mirrors the verification commands that run locally)

## Files

- Create: `.github/workflows/ci.yml`

## Required Changes

1. **Create `.github/workflows/ci.yml`** with these properties:

   - **Workflow name:** `CI`
   - **Triggers:**
     - `push` on the `main` branch
     - `pull_request` on the `main` branch
     - `workflow_dispatch` (allow manual runs)
   - **Concurrency:** group by `ci-${{ github.ref }}`; cancel in-progress runs for the same branch when a new commit lands.
   - **One job named `verify`** running on `macos-15` (Apple Silicon runner — matches the scaffold's macOS 14+ deployment target and provides Swift 6.x). Steps in order:
     1. `actions/checkout@v4`
     2. Select Xcode 16 (or the latest Xcode that ships Swift 6.x) via `sudo xcode-select -s /Applications/Xcode_16.app`. If Xcode 16 isn't pre-installed on the runner, fall back to the default Xcode and emit a clear log line naming the Swift version in use (run `swift --version`).
     3. `brew install swift-format swiftlint` — verify both are on PATH after install with `swift-format --version` and `swiftlint version`.
     4. Cache the SwiftPM build directory (`.build/`) keyed on `Package.resolved`'s hash, with a fallback restore key.
     5. `swift-format lint --recursive --strict Sources Tests`
     6. `swiftlint lint --strict --quiet`
     7. `swift build`
     8. `swift test`

2. **Use action versions pinned to a major release** (`@v4`, not `@main`). Where a third-party action is needed (caching), prefer `actions/cache@v4`. Do not introduce custom action references.

3. **Do not** introduce a coverage check in this workflow. The 90% coverage floor lands in ho-01 when there's real code to measure against. Keep this CI minimal.

4. **Verification step output** should be readable. Don't use `--quiet` on `swift build`. Use `--strict` and `--quiet` on `swiftlint` as in the local pre-commit hook.

## Acceptance

- [ ] `.github/workflows/ci.yml` exists at the documented path.
- [ ] The workflow triggers on push-to-main, PR-to-main, and manual dispatch.
- [ ] The single `verify` job runs on `macos-15`.
- [ ] All four verification commands (swift-format lint, swiftlint, swift build, swift test) appear as named steps in the documented order.
- [ ] Action versions are pinned to a major release tag.
- [ ] SwiftPM `.build/` is cached keyed on `Package.resolved`.
- [ ] No coverage step is present (that's ho-01's work).
- [ ] The workflow file passes `yamllint` (or equivalent YAML validity check) and `actionlint` if available locally.
- [ ] Pushing this commit triggers the workflow on GitHub and the workflow succeeds.

## Verification

```bash
# Local YAML validity
yamllint .github/workflows/ci.yml 2>/dev/null || python3 -c "import yaml; yaml.safe_load(open('.github/workflows/ci.yml'))"

# actionlint if installed (brew install actionlint)
actionlint .github/workflows/ci.yml 2>/dev/null || echo "actionlint not installed — skipping"

# Commit + push, then watch the run
git add .github/workflows/ci.yml
git commit -m "ho-00: add GitHub Actions CI workflow (Ho-00-AT-01)"
git push

# Watch the run via gh CLI (assumes you're authenticated for sageframe-no-kaji)
gh run watch --exit-status

# Confirm green
gh run list --limit 3
```

The acceptance "workflow succeeds on push" is the live test. If it fails, fix the workflow (toolchain selection, install order, etc.) before declaring done.

## Do Not

- Do not add coverage measurement. The 90% floor lands in ho-01.
- Do not skip the SwiftPM build cache. Cold builds take >2 minutes; caching matters for fast feedback.
- Do not use `--no-fail-fast` or any flag that masks test failures.
- Do not pin to `macos-latest` — that label drifts. Pin to `macos-15` explicitly.
- Do not add a matrix over Swift versions or macOS versions. The project's verification stack runs against the toolchain documented in `~/.claude/modules/languages-swift.md` (Swift 6.0+) on the canonical runner. Matrix expansion is a later optimization, not a v1 concern.

## Stop Condition

If the `macos-15` runner ships a Swift toolchain older than 6.0 by default and selecting Xcode 16 doesn't fix it, stop and surface the finding before working around it. The project's `Package.swift` declares `swift-tools-version: 6.0` and `swiftLanguageModes: [.v6]`; a runner that can't satisfy that is a real constraint that needs the practitioner's input (likely: install Swift via `swiftly`, or adjust the runner image).

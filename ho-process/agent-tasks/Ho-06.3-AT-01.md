# Ho-06.3-AT-01 — First-run wizard + GUI key bootstrap

**Goal**

Give the Workshop its front door: when `vaultState == .noVault`, the main
window renders a paged first-run wizard instead of the bare empty state —
prerequisite check (`age`/`age-keygen`), age key generate-or-import into the
Keychain, a verified backup nudge, initial scan-root detection, an optional
git remote — ending in a created vault and the model flipped to `.open`. All
app-layer; the Keychain **write** capability the GUI has never had arrives
here, mirroring the CLI's bootstrap.

**Context**

ho-06.3 Decisions 1–5 and 8 (`ho-process/hos/ho-06.3-first-run-and-ingest.md`).
Everything Core-side already exists: `VaultCore.createVault(at:)`
(`Sources/SharibakoCore/VaultCore.swift:32`), `Conduit.initializeRepository()`
/ `setRemote(_:)` / `setIdentity(name:email:)` (`Sources/SharibakoCore/Conduit.swift`),
`WorkshopConfig.persistScanRoot` (`Sources/Sharibako/Support/WorkshopConfig.swift`).

  file → Keychain store → scrub; recipient from the `# public key:` header.

**Required Changes**

1. **`GUIShell`.** `findExecutable(_ name:)` honoring PATH first, then the
   same fixed fallback directories Core's `Shell` documents (Homebrew paths
   included); `run(_ url:, _ args:)` returning exit code / stdout / stderr.
   Mirror, don't copy commentary wholesale.

2. **`GUIAgeKeyBootstrap`.** Four capabilities, each testable through an
   injected store/probe seam so no test touches the real Keychain:
   - `prerequisitesPresent()` — both `age` and `age-keygen` resolvable.
   - `generateToKeychain()` — `age-keygen -o <temp 0600>` → store bytes under
     `.userPresence` (delete-then-add, the CLI's exact attributes) → scrub +
     delete temp → return `(identity: String, recipient: String)` held only
     long enough for the backup page.
   - `importToKeychain(from url:)` — validate the file contains an
     `AGE-SECRET-KEY-1` line (reject otherwise with a typed error), store the
     same way; derive the recipient via `age-keygen -y <file>` (hand-made
     identity files have no `# public key:` header).
   - `keychainKeyExists()` — the CLI probe's non-authenticating query,
     mirrored.

3. **First-run state machine in `WorkshopModel+FirstRun`.** An observable
   page progression (welcome/prereq → key → backup → root → remote → finish)
   with tested guards:
   - Prereq page blocks until `prerequisitesPresent()`; exposes the
     copy-paste `brew install age` line and a re-check intent.
   - Key page: if `keychainKeyExists()` → state says so and skips generation
     (never a second key); else generate or import per Decision 1. Import
     skips the backup page (Decision 2 — the user's identity file already
     lives outside the Keychain).
   - Backup page (generate path only): holds the fresh identity + recipient;
     `verifyBackup(at url:)` re-reads the saved file and passes only when its
     contents match the stored identity; Continue is disabled until verified.
     After the page advances, the in-memory identity is cleared.
   - Root page: `suggestScanRoot(home:)` probes `~/Projects`, `~/Developer`,
     `~/Code`, `~/Vaults`, `~/src`, `~/dev` — existing directories only,
     tie-broken by which contains git repositories (shallow check); pure
     logic over an injected home. The page states the vault location plainly
     (`~/.sharibako/vault`, informative only — Decision 3).
   - Remote page: optional URL string; empty = skip.
   - `completeFirstRun()`: `VaultCore.createVault` at the resolved vault URL
     → `Conduit.initializeRepository()` → git-identity probe (via `GUIShell`
     `git -C <vault> config user.email`; empty → `setIdentity("Sharibako",
     "sharibako@localhost")`, Decision 8) → `setRemote` when given (surface
     `remoteURLRejected` inline on the remote page, don't crash) →
     `persistScanRoot` → flip `vaultState` to `.open` → `loadScopes()` + the
     launch-scan path 06.1 built. Announce the outcome on the status surface
     (the silent-success rule).

4. **`FirstRunWizard` view.** Declarative pages over the tested state, calm
   pālana grounds (`ground`/`groundDeep` tokens — no raw colors, no hex),
   save panel via `NSSavePanel` (suggested name `sharibako-key-backup.txt`),
   folder picker for the root override, plain-talk copy that names stakes
   ("lose the key, lose the vault"). No time bars, no decoration — nothing on
   screen that isn't working.

5. **Wiring.** `WorkshopWindow`'s `.noVault` arm renders the wizard. The
   finish hand-off seam for AT-02 (the ingest invite) is a model-side hook —
   e.g. `firstRunCompleted` state AT-02 consumes; leave a named seam, don't
   build the invite here.

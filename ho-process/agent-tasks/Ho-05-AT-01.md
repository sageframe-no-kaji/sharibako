## Required Changes

### 1. Xcode project (`xcode/Sharibako.xcodeproj`)

- An app target `Sharibako` whose product is `Sharibako.app`, deployment target
  macOS 14, that references the local package at the repo root as a Local
  Package Reference and links the `SharibakoCore` **library product**.
- The app compiles the `Sources/Sharibako/**` Swift files. The SwiftPM
  `Sharibako` **executable** target stays in `Package.swift` as a headless
  compile-check (CI keeps building the app sources without Xcode) — keep both in
  sync on the same source directory.
- Info.plist: bundle id `net.sageframe.sharibako` (or the app-specific id from
  palana), `LSMinimumSystemVersion` 14.0, a placeholder app name/icon.
- Entitlements: `keychain-access-groups` including the shared group so the app
  reads the CLI-written age-key item.
- Signing set to the Developer ID / team from palana. Notarization is **not**
  wired here (that's ho-08) — this is local signed-run capability only.

### 2. GUI Keychain adapter (`Support/GUIAgeKeyProvider.swift`)

Mirror `KeychainAgeKeyProvider.loadIdentity` and the `FileAgeKeyProvider` shape
into the GUI target (Decision 1 — copy the pattern, do not import CLI code):

- A `GUIAgeKeyProvider` protocol (or a single enum with two cases) exposing
  `loadIdentity(reason:) throws -> AgeKeyHandle`, where `AgeKeyHandle` wraps a
  key-file URL + a `release()` cleanup closure (mirror the CLI's `AgeKeyHandle`).
- **Keychain path** (`#if os(macOS)`): `SecItemCopyMatching` with an `LAContext`
  carrying `reason`, service `sharibako`, account `sharibako.age-key`, access
  group `3N8F759K8D.net.sageframe.sharibako`; write the returned bytes to a
  `0600` temp file; `release()` best-effort scrubs and deletes it. The CLI's
  `TempKeySignalGuard` is **not** ported — the GUI is not a signal-driven CLI;
  a plain `defer`/`release()` scrub is the whole cleanup.
- **File path**: a provider that hands back a caller-supplied age-key file URL
  directly with a no-op `release()` — the dev/test bypass (Decision 7). The
  model selects this when a dev key path is configured (see change 4).
- This file is **coverage-excluded** (dogfood-only, like the CLI's provider) —
  add it to ci.yml's `EXCLUDED` with a comment.

### 3. Config resolution (`Support/WorkshopConfig.swift`)

- `resolveVaultURL(environment:home:)` mirroring `VaultLocator` precedence minus
  the flag: `SHARIBAKO_VAULT` env → `~/.sharibako/vault/`. Pure and injectable
  (env + home as parameters defaulting to live values) so tests exercise every
  branch. Does **not** create anything.
- `loadScanRoots(configURL:) -> [URL]` reading `scan_roots` from
  `~/Library/Application Support/Sharibako/config.yaml` (Yams) when present, else
  `[]`. `persistScanRoot(_:configURL:)` writing a chosen root back. The actual
  `NSOpenPanel` invocation lives in AT-03's Rescan; here just the read/write +
  resolution helpers, unit-tested against a temp config file.

### 4. `WorkshopModel` (`WorkshopModel.swift`)

- `@Observable @MainActor final class WorkshopModel`.
- State: `vaultState` enum (`.noVault(expectedPath: URL)` / `.open(vaultURL:
  URL)`), `scopes: [ScopeMetadata]`, `selectedScopeID: String?`, an
  `errorMessage: String?` surface.
- `init` resolves the vault URL via `WorkshopConfig`; if it's an existing vault
  directory, sets `.open` and loads scopes, else `.noVault`.
- `loadScopes()` calls `VaultCore(vaultURL:).listScopes()` and sorts/stores;
  maps thrown `VaultError` to `errorMessage`.
- A `devAgeKeyPath: URL?` set from a `SHARIBAKO_AGE_KEY` env read (dev bypass) —
  used by AT-02/03 to pick the file provider over Keychain. Wire the field and
  the provider selection here; the reveal call itself is AT-02.
- All Core-touching methods are `@MainActor` and do their (fast, local)
  filesystem work inline; keep them synchronous in v1 (no `async` yet — the
  vault ops are local and quick).

### 5. Views (`WorkshopWindow.swift`, `ScopeSidebar.swift`) and `App.swift`

- `WorkshopWindow`: a `NavigationSplitView` with three columns — sidebar
  (`ScopeSidebar`), a center placeholder ("select a scope" — filled in AT-02),
  and a detail placeholder (AT-02). Renders the `.noVault` empty state (naming
  the expected path) when the model says so.
- `ScopeSidebar`: sections per `ScopeType` (Projects-dev, Projects-prod,
  Services, Machines, Other), each listing its scopes by `displayName ??
  identity`, bound to `selectedScopeID`.
- `App.swift`: construct one `WorkshopModel`, inject via `.environment`, host
  `WorkshopWindow` in the `WindowGroup`. Remove the placeholder `ContentView`.
- All three `View` files are coverage-excluded (add to ci.yml `EXCLUDED`).

### 6. Tests (`WorkshopModelTests.swift`)

swift-testing `@Suite`. Drive the model and config helpers with temp-directory
vaults and injected env/home — no Keychain, no signing:

- `resolveVaultURL` honours `SHARIBAKO_VAULT`, falls back to the default.
- A model pointed at a populated temp vault loads and groups scopes; a model
  pointed at an empty/absent path lands in `.noVault` with the expected path.
- `loadScanRoots` / `persistScanRoot` round-trip through a temp config file.
- A thrown `VaultError` surfaces as `errorMessage`, not a crash.

  keep v1 synchronous on `@MainActor`.

## Stop Condition

If generating `xcode/Sharibako.xcodeproj` cleanly requires adding a build-tool
dependency (XcodeGen, Tuist) rather than committing an Xcode-authored project,
**stop and surface the choice** before adding it — a new toolchain dependency is
the practitioner's call, not a silent default. A committed, Xcode-authored
`.xcodeproj` is the preferred path if it's reproducible.

# Ho-06.3-AT-02 — GUI ingest flow

**Goal**

Bring the repos-to-vault journey into the Workshop: an ingest sheet that
scans a chosen directory, presents each detected secret with the five CLI
verdicts (import as scope-local / link to shared / move to shared / leave
alone / skip), confirms scope ID and type, and commits through the existing
`Materializer.acceptIngest` behind one Touch ID. Three entry points share the
one surface: an **Ingest Project…** panel verb, the first-run wizard's finish
hand-off (seeded with detected `.env`-bearing candidates), and — in AT-03 —
orphaned-marker rows.

**Context**

ho-06.3 Decisions 1 (page 6) and 6 (`ho-process/hos/ho-06.3-first-run-and-ingest.md`).
The Core API is complete and unchanged: `Materializer.ingest(directory:)`
returns a `ProposedScope` (keyless — parses `.env`/`.env.local`/`.env.example`
and reads the vault filesystem), `acceptIngest(_:decisions:scopeID:scopeType:)`
encrypts and drops the `.sharibako` marker
(`Sources/SharibakoCore/Materializer+Ingest.swift`). The CLI's `InitCommand`
(`Sources/SharibakoCLI/Commands/InitCommand.swift`) is the behavioral
reference: nothing-to-import rejection, re-init reconcile (present only keys
the scope doesn't own — see its `filterProposal`), scope-ID collision
confirmation, suggested ID/type defaults. Mirror the semantics in
`WorkshopModel`; do not import CLI code.

Existing machinery to ride: `VaultWorker` for the scan/commit hops (ho-06.1 —
nothing beach-balls the main actor), `makeAgeKeyProvider()` + the 5-minute
Touch ID reuse window for the encrypt, the status surface for announcements,
the launch-scan cache refresh after commit (glyphs must be right immediately —
06.2 reads what ingest writes).

**Required Changes**

1. **Ingest session state in `WorkshopModel+Ingest`.** `beginIngest(directory:)`
   hops to the worker, runs keyless `Materializer.ingest`, and lands a session:
   proposal, per-key decision selections (default `.importAsLocal`), scope ID
   (suggested, editable, validated by `VaultCore.isValidIdentifier`), scope
   type (suggested). Reconcile semantics on a directory that already has a
   marker: only keys the scope doesn't own are presented (mirror
   `filterProposal` via `inspect`); nothing new → announce "already
   initialized, nothing to reconcile" and no sheet. No importable secrets →
   announce plainly, no sheet (never an empty scope). Scope-ID collision with
   a different existing scope → the sheet says keys will be *added* to it
   (the CLI's confirm, rendered inline).
2. **Decision rows.** Each detected key: masked value preview, the five
   verdicts — `linkToShared`/`moveToShared` offered only when meaningful
   (link requires existing shared entries; both target an ID). Keys from
   `.env.example` without values (`suggestedKeysNeedingValues`) are listed
   informationally ("no value found — add later with Add Secret"), not
   decision rows. Parse warnings surface, not swallowed.
3. **Commit.** `commitIngest()` acquires the age key on the main actor
   (existing provider, reason "Encrypt secrets during ingest"), hops to the
   worker for `acceptIngest`, reloads scopes + scan cache, selects the new
   scope in the sidebar, announces the CLI's summary shape ("imported N,
   linked N, moved N, left alone N, skipped N — marker at <path>"). Failure
   announces and leaves the sheet up (decisions intact). Cancel discards.
4. **Panel verb.** **Ingest Project…** in the create group, always-titled,
   enabled when the vault is open and no activity is in flight; directory
   picker → `beginIngest`.
5. **Wizard hand-off.** On `completeFirstRun`, a worker-side shallow walk of
   the scan root finds `.env`-bearing directories (bounded depth, mirroring
   the marker scan's discipline — fast on a wide root). Candidates → the
   finish page lists them, one tap starts ingest on that directory. None →
   "you're set up" close. The walk logic is tested; the invite is view-layer.

## Required Changes

### 1. `VaultCore.updateNotes(_:inScope:notes:)` (Decision 6)

- In `VaultCore+Encryption.swift`, next to `rotate`. Reads the existing
  `<KEY>.age`, and re-encrypts a `SecretContent` with the **same `value`, same
  `rotatedAt`**, and the new `notes`. A notes-only edit is not a rotation and
  must not bump `rotated_at`.
- Throws `VaultError.secretNotFound(scope:key:)` when the `.age` is absent
  (mirror `rotate`). Only valid for `.value` secrets — a `.link` has no local
  notes; surface that as `secretNotFound` (no `.age` at the key) or a clear
  error, do not silently write.

### 2. `WorkshopModel` — mutation + action intents

Each intent runs on `@MainActor`, calls the Core op, refreshes the affected
state (scope list / secret list / reveal), and maps thrown `VaultError` to
`errorMessage`:

- `addScope(id:type:displayName:)` → `createScope`; refresh sidebar, select it.
- `addSecret(key:value:notes:inScope:)` → `addSecret`; refresh the secret list.
- `addSharedEntry(id:value:notes:)` → `addSharedEntry`.
- `editValue(key:inScope:newValue:)` → `rotate`; clear any revealed value for
  that key (force re-reveal of the new value).
- `editNotes(key:inScope:notes:)` → `updateNotes`.
- `materializeSelectedScope()` → resolve the marker via
  `Materializer.resolveMarker(forScope:scanRoots:)` using the resolved scan
  roots, then `materialize(...)`. On `MaterializeResult.diffPending`, surface the
  diff and require an explicit "overwrite" confirmation before re-calling with
  the overwrite flag (mirror the CLI's `--force` gate — do not overwrite drift
  silently).
- `sync()` → `Conduit.commit` then `push`; a no-remote vault is a clean no-op,
  a rejected push / conflict maps to `errorMessage` (mirror the CLI's mapping).
- `rescan()` → open an `NSOpenPanel` directory picker when no scan root is
  configured, persist the choice via `WorkshopConfig.persistScanRoot`, then
  `Materializer.scan(roots:)`; store the result for materialize resolution.

### 3. Views

- Three sheets (`AddScopeSheet`, `AddSecretSheet`, `AddSharedEntrySheet`) with
  validated fields — use `VaultCore.isValidIdentifier` for scope/shared IDs and
  keys, disabling submit on invalid input. Values entered with a secure field.
- `SecretDetail` gains an Edit affordance: edit the value (submits via
  `editValue` → `rotate`) and edit notes (submits via `editNotes` →
  `updateNotes`). Editing value and notes are distinct submits so a notes edit
  never rotates the value.
- `WorkshopWindow` toolbar: Add Scope, Add Shared Secret, Materialize (enabled
  when a scope is selected), Sync, Rescan. A materialize-drift confirmation
  dialog surfaces the diff.
- All new/modified pure-`View` files coverage-excluded; the confirmation/branch
  logic lives in `WorkshopModel` (tested), not the views.

### 4. Tests

- `updateNotes`: encrypt a secret with value V, notes N, rotatedAt R; call
  `updateNotes` with notes N'; assert value == V, rotatedAt == R, notes == N'.
  Absent secret throws `secretNotFound`.
- `WorkshopModel` (file-key path, temp vault; a bare remote for the sync leg):
  - `addScope` / `addSecret` / `addSharedEntry` land in the vault and refresh
    state;
  - `editValue` rotates (value changes, rotatedAt bumps) and clears the stale
    reveal; `editNotes` changes notes without bumping rotatedAt;
  - `materializeSelectedScope` writes the `.env` at the marker target and
    returns `diffPending` (not a silent overwrite) when drift exists;
  - `sync` commits and no-ops cleanly without a remote; maps a forced conflict
    to `errorMessage`.

**Dogfood gate (signed install + Touch ID — the only proof of the real path):**

1. Build and sign `Sharibako.app` (identity from palana), install it.
2. Open the existing dogfooded vault (Keychain path, not the file-key bypass).
3. Reveal a real secret via Touch ID; confirm the shared-item claim — a value
   `sharibako get` can read reveals in the Workshop with no re-keying.
4. Add a scope + secret, edit a value (rotate) and a note, Materialize the
   scope, and confirm the written `.env` matches `sharibako materialize` for the
   same scope. Sync.

## Stop Condition

entitlement/access-group mismatch), stop and surface before working around it:
Decision 1's shared-item claim is load-bearing, and a mismatch is an
architecture finding, not a view-layer fix.

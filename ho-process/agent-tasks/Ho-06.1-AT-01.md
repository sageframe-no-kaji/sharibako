# Ho-06.1-AT-01 — Concurrency foundation: `VaultWorker`, async intents, scan cache

**Goal**

Move the Workshop's tree-walking and network operations (`rescan`,
`materializeSelectedScope`, `sync`) off the main thread through a new
`VaultWorker` actor, with a published `activity` state driving progress UI,
and give `WorkshopModel` an in-memory scan-result cache populated by a
non-blocking launch scan so materialize stops re-walking the scan root per
action. The window stays interactive during every long operation.

**Context**

ho-05 Decision 2 ratified synchronous-on-`@MainActor`; the dogfood gate proved
it false for scans (main-thread beach ball) — ho-06.1 Decision 1 amends it.
The amendment is bounded: `WorkshopModel` stays `@Observable @MainActor` and
owns all published state; only the blocking work hops to the actor.
`Materializer` and `VaultCore` are `Sendable` structs and cross without Core
changes. Serialization (one vault operation at a time — what the synchronous
posture gave for free) must be preserved, and the actor provides it by
construction. Touch ID / age-key acquisition stays on the main actor; only the
`AgeKeyHandle` (`Sendable`) crosses. Fast single-file operations (reveal, add,
rotate, notes) stay synchronous — ho-05's Reflect confirmed that premise
holds. Decisions 1–2 in
`ho-process/hos/ho-06.1-workshop-responsiveness.md`.

**Required Changes**

1. **`VaultWorker` actor** in `Sources/Sharibako/Support/VaultWorker.swift`.
   A minimal actor whose job is executing `Sendable` closures on its own
   executor, serially by construction:

   - `func run<T: Sendable>(_ work: @Sendable () throws -> T) rethrows -> T`
     (or the equivalent shape strict concurrency accepts cleanly).
   - No vault knowledge, no state beyond what serial execution needs. Keep it
     small; its value is the isolation domain, not logic.

2. **Async intents on `WorkshopModel`.** `rescan`, `materializeSelectedScope`
   (both the initial and `force:` paths), and `sync` become `async`. Shape per
   intent: set `activity`, acquire the age key on main where the operation
   needs one (materialize), `await worker.run { ... }` for the blocking Core
   work, publish results back on the main actor, clear `activity` on every
   exit path (`defer`). `handle.release()` runs after the awaited work
   completes, on every path. Error mapping to `errorMessage` and the existing
   `statusMessage` vocabulary are unchanged — this task changes *where* work
   runs, not what it says.

3. **`activity` published state.** An optional enum on `WorkshopModel` naming
   the operation in flight (scanning / materializing / syncing) with display
   text. Non-nil `activity` means: toolbar action buttons that trigger vault
   operations are disabled, and the status surface shows a progress indicator
   with the activity's text. Intents guard against re-entry (an intent called
   while `activity` is non-nil returns without doing work) — the actor
   serializes execution, but the guard keeps the UI honest about it.

4. **Scan cache.** `WorkshopModel` holds the latest scan report
   (`private(set)`, main-actor state). Populated by:

   - **Launch scan:** a new `performLaunchScan()` async intent, called from a
     `.task` modifier on the window (SwiftUI-idiomatic view-scoped work). No-op
     when `scanRoots` is empty or the vault is not open. Runs through the
     worker; the window renders immediately and the cache fills in behind it.
     Quiet on success at launch (no `statusMessage` — the user didn't trigger
     it); failures still land in `errorMessage`.
   - **Rescan:** the existing intent now also stores its report in the cache
     (its user-facing announce is unchanged).

5. **Cache-based marker resolution.** `materializeSelectedScope` resolves the
   selected scope's marker from the cache instead of calling
   `resolveMarker(forScope:scanRoots:)` (which re-walks the root). On a cache
   miss for the scope: run one fresh scan through the worker, update the
   cache, retry the lookup; only then surface the marker-not-found error. The
   jump-to-directory button (AT-02) and ho-06.2's glyphs read this same cache
   — expose a lookup like `cachedMarker(forScope:) -> ScopeMarker?`.

6. **Tests.** Async intent tests (swift-testing is async-native): the cache
   populates from launch scan and refreshes on rescan; materialize resolves
   from cache without re-scanning (observable via a temp-root fixture — e.g.
   delete the root after the cache is warm and confirm materialize still
   resolves); the miss-fallback runs exactly one fresh scan; re-entry guard
   (second intent while `activity` non-nil no-ops); `activity` clears on error
   paths. All tests use injected homes/roots/temp vaults — never live user
   state.

**Do Not**

- Do not move `WorkshopModel` off `@MainActor` or change its `@Observable`
  posture — Decision 1 amends where work runs, not the state model.
- Do not make reveal/add/rotate/notes async — the synchronous premise holds
  for single-file operations.
- Do not persist the scan cache to disk (Decision 2: a persisted cache lies
  across sessions).
- Do not use ad-hoc `Task {}` / `Task.detached` in intent bodies as the
  offload mechanism — the worker is the isolation domain.
- Do not modify `SharibakoCore` — this task needs no Core changes
  (`Materializer`/`VaultCore` are already `Sendable`).

**Stop Condition**

If strict concurrency forces changes to `SharibakoCore` beyond what already
compiles (e.g. a type turns out not to be `Sendable` in practice), stop and
surface before touching Core — Core changes in this ho are AT-02/AT-03's two
named additions only.

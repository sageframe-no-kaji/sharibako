# Ho-06.7-AT-03 — GUI: Delete Scope in the action panel

**Goal**

Add a Delete Scope affordance to the Workshop's action panel: a destructive
button in the scope group, toned with the pālana rust token, disabled when no
scope is selected, opening a system confirmation dialog that names what will be
removed. On confirm it calls a tested `WorkshopModel` delete intent that removes
the scope via `VaultCore.deleteScope` and refreshes the sidebar. This is the
first real in-window consumer of `Color.drift`.

**Context**

ho-06.7 Decision 5 (`ho-process/hos/ho-06.7-delete-verb.md`). Deletion is new
vault behavior, but the GUI wiring mirrors the existing panel verbs: branching
logic lives in `WorkshopModel` (tested), the View body stays declarative and is
CI-excluded. The ho-06.5 gate settled the rust boundary empirically: an
**in-window button can wear `Color.drift`**, but a **confirmation dialog cannot**
(dialogs render out of the view hierarchy and take the system's own destructive
red) — so the button reads rust and the dialog stays fully system-rendered. Do
not attempt to tint the dialog; that path was tried twice at the 06.5 gate and
does not work. `deleteScope` is keyless (AT-01) — no Touch ID.

Follow the panel's established shape: `Sources/Sharibako/Views/ActionPanel.swift`
`scopeGroup` (built in ho-06.5) is where selected-scope verbs live (Materialize,
Preview .env, Jump to Directory). Mirror whichever existing verb best shows the
model-intent + disabled-state + activity pattern (`selectedScopeID`,
`model.activity`).

**Scope note:** this task adds **Delete Scope only**. There is no shared-entry
list in the Workshop yet (the shared-secret browser is ho-07), so the GUI
delete-shared affordance defers there. Core and CLI (AT-01/AT-02) carry
shared-delete already.

**Files**

- Modify: `Sources/Sharibako/WorkshopModel.swift` (or a `WorkshopModel+…`
  extension if the file nears the SwiftLint `file_length` ceiling, per the
  established precedent) — a `deleteSelectedScope()` / `deleteScope(id:)` intent.
- Modify: `Sources/Sharibako/Views/ActionPanel.swift` — the Delete Scope button
  in `scopeGroup`, rust-toned, with the confirmation dialog.
- Modify: the WorkshopModel test target (`Tests/SharibakoTests/`) — unit tests
  for the delete intent over injected temp vaults.
- Modify: `.github/workflows/ci.yml` — only if a new View file is added; join the
  named-`EXCLUDED` coverage regex with a justification comment.

**Required Changes**

1. **Delete intent in `WorkshopModel`.** A method that deletes a scope by id
   through `VaultCore.deleteScope`, mirroring how existing intents construct the
   vault, set `activity`, handle errors onto the status surface, and refresh
   state. After a successful delete: reload the scope list (the same reload other
   mutating intents use) and clear `selectedScopeID` if it pointed at the deleted
   scope. Announce the outcome on the status surface (the silent-success rule —
   name the deleted scope). On failure, announce the error; do not crash. No age
   key, no Touch ID.

2. **Delete Scope button** in `ActionPanel.scopeGroup`. An always-titled button
   labeled "Delete Scope", placed in the selected-scope group, with:
   - `Color.drift` as its tint/foreground (the rust voice — the destructive
     in-window affordance Decision 5 names; wire it the way the other titled
     panel buttons carry color).
   - Disabled when `selectedScopeID == nil` or `model.activity` indicates a
     run in flight (mirror the sibling verbs' disabled logic).
   - A `.confirmationDialog` (or `.alert`) presenting a **system-rendered**
     destructive confirm — `role: .destructive` on the confirm button, **no
     tint** — titled/messaged to name the scope and the blast radius ("Delete
     scope "<name>" and all its secrets? Materialized .env files and markers
     are left in place. The removal is committed on next Sync."). Cancel is the
     default. Confirm calls the WorkshopModel intent from change 1.

3. **Tests** for the delete intent over injected temp vaults (no live user
   state): deleting a scope removes it from the model's reloaded scope list and
   clears the selection when it matched; deleting surfaces the outcome on the
   status surface; a failure path (e.g. already-absent) announces an error and
   leaves the model consistent. The button itself (View body) is not unit-tested;
   it is CI-excluded — the signed-install gate covers it.

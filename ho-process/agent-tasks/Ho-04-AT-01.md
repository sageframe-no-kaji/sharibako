# Ho-04-AT-01 — CLI scaffold, Keychain, key management, inspection verbs

## Goal

Build the `SharibakoCLI` foundation: Swift ArgumentParser subcommand tree with global flags (`--vault`, `--age-key`, `--json`), the `AgeKeyProvider` protocol with both implementations (`KeychainAgeKeyProvider` for macOS, `FileAgeKeyProvider` for Linux and tests), the `sharibako key generate | import | export` commands, and the four inspection verbs (`status`, `scan`, `list`, `heal`). Add a repo-root `scripts/install.sh` that builds and installs the binary. Establish the exit code taxonomy, error rendering to stderr with actionable remediation, and human/JSON output rendering as reusable helpers. Full test coverage for everything not gated by real biometry, with headless-safe Keychain code paths behind the `AgeKeyProvider` seam.

### 2. `AgeKeyProvider.swift` — protocol + File impl

```swift
public protocol AgeKeyProvider: Sendable {
    /// Returns the URL of an age identity file usable with `age --identity <url>`.
    /// Callers own the returned URL for the duration of the operation; providers may
    /// return a temp file whose contents are scrubbed and deleted on cleanup.
    func loadIdentity(reason: String) throws -> AgeKeyHandle
}

public struct AgeKeyHandle {
    public let url: URL
    private let cleanup: () -> Void
    public init(url: URL, cleanup: @escaping () -> Void) { … }
    public func release() { cleanup() }
}

public struct FileAgeKeyProvider: AgeKeyProvider {
    public let path: URL
    public init(path: URL) { self.path = path }
    public func loadIdentity(reason: String) throws -> AgeKeyHandle {
        // No copying — just hand back the file's URL. No-op cleanup.
    }
}
```

`reason: String` argument surfaces to the Touch ID prompt on macOS. Ignored by `FileAgeKeyProvider`.

### 3. `KeychainAgeKeyProvider.swift` — macOS-only, `#if os(macOS)`

`SecItemCopyMatching` with a query that:
- `kSecClass = kSecClassGenericPassword`
- `kSecAttrService = "sharibako"`
- `kSecAttrAccount = "sharibako.age-key"`
- `kSecReturnData = true`
- `kSecUseAuthenticationContext = LAContext()` — carries the `localizedReason` passed as `reason:`

The returned `CFData` is decoded to `Data`, written to a securely-created temp file (`FileManager.default.temporaryDirectory` + `UUID`, permissions `0600`), and returned as `AgeKeyHandle`. The `cleanup` closure `memset_s`'s the underlying buffer (best-effort via `Data.withUnsafeMutableBytes`) and removes the temp file.

Also expose:
- `func storeIdentity(_ contents: Data) throws` — write to Keychain with `SecAccessControl` requiring `.userPresence`. Used by `key generate` and `key import`.
- `func exportIdentity(reason: String) throws -> Data` — retrieve the raw private-key contents. Used by `key export --private`.

### 6. `ErrorReporter.swift`

Central mapping from `VaultError` → `(SharibakoExitCode, humanMessage, remediation)`.

```swift
struct ErrorReport {
    let code: SharibakoExitCode
    let message: String
    let remediation: String?
}

enum ErrorReporter {
    static func report(_ error: Error, json: Bool) -> Never {
        // Print to stderr, exit with the right code.
    }
}
```

Every `VaultError` case gets a specific mapping. Some examples:

- `.vaultNotFound(path:)` → `.filesystem`, "Vault not found at `<path>`.", "Run `sharibako key generate` to create one."
- `.scopeNotFound(id:)` → `.userError`, "Scope `<id>` does not exist.", nil.
- `.markerNotFound(startingFrom:)` → `.userError`, "No `.sharibako` marker found starting from `<path>`.", "Run `sharibako init` in your project directory to create one."
- `.ageInvocationFailed(exitCode:stderr:)` → `.age`, "age binary failed (exit `<code>`): `<stderr>`.", "Verify `age` is installed and on PATH."
- `.shellNotFound(name:)` → `.age` or `.git` depending on name, "The `<name>` binary was not found on PATH.", "Install `<name>` (`brew install age`, etc.)."
- Every git failure → `.git`.

### 8. `KeyCommand.swift`

Parent `key` subcommand with three children:

- `sharibako key generate`
  - Shell out to `age-keygen`, capture output (private key + `# public key:` header).
  - On macOS: store private key in Keychain via `KeychainAgeKeyProvider.storeIdentity`. Print public key to stdout with "Save this recipient key somewhere safe:" preamble on stderr.
  - On Linux: prompt for a passphrase interactively (via `readpassphrase`-style), run `age -p` to encrypt the raw private key, write to `~/.config/sharibako/age-key`, chmod `0600`. Print public key + backup nudge to stderr.
  - `--force` overrides an existing key (with confirmation prompt unless `--yes`).
  - Refuses to overwrite silently.

- `sharibako key import <path>`
  - Read the file at `<path>`, validate it's an age key (basic `AGE-SECRET-KEY-` prefix check or plugin-identity check).
  - Store into Keychain (macOS) or copy to `~/.config/sharibako/age-key` (Linux).
  - Ask: "Delete the source file? [y/N]" — respect `--delete-source` / `--keep-source` to skip the prompt.

- `sharibako key export [--public | --private --i-know-this-is-plaintext] [--recipient-only]`
  - `--public` (default): retrieve identity, extract the `# public key:` header line, print to stdout.
  - `--private --i-know-this-is-plaintext`: retrieve identity, print raw contents to stdout. Requires both flags together (specific belt-and-suspenders wording is a decision the practitioner can revise if it feels wrong).

### 9. Inspection subcommands (thin wrappers over `SharibakoCore`)

**`StatusCommand`**: constructs a `VaultCore(vaultURL:)` (no age key needed), calls `listScopes()`, `listShared()`, optionally `linkGraph()` if a specific scope is given. Renders as table (human) or JSON. Optional `<scope>` arg limits output to one scope's `inspect(scopeID:)` result.

**`ScanCommand`**: takes an optional `<root>` (defaults to cwd), constructs a `Materializer(vaultCore:, vaultURL:)`, calls `scan(roots: [root])`. Renders each marker's absolute path + scope name. JSON output emits `[{path, scope, target}, ...]`.

**`ListCommand`**: `--shared` flag toggles between `listScopes()` (default) and `listShared()`. One column output human, JSON array.

**`HealCommand`**: takes optional `<scope>`. Resolves scope from cwd via `Materializer.resolveMarker(startingFrom:)` when absent. Loads `AgeKeyProvider`, opens `VaultCore(vaultURL:ageKeyURL:)` with the identity file, constructs `Materializer`, calls `heal(marker:)`. Renders the `DriftReport`: `.match` in green, `.fileValueDiffers` in yellow with SHAs, `.fileMissing` in red. JSON emits the `DriftReport` structure verbatim.

### 10. `scripts/install.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail

REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
DEST="${SHARIBAKO_INSTALL_DIR:-/usr/local/bin}"

cd "$REPO_ROOT"
echo "Building sharibako (release)..."
swift build -c release --product sharibako
BINARY_PATH="$REPO_ROOT/.build/release/sharibako"

if [[ ! -f "$BINARY_PATH" ]]; then
    echo "Error: build succeeded but binary not found at $BINARY_PATH" >&2
    exit 1
fi

mkdir -p "$DEST"
install -m 0755 "$BINARY_PATH" "$DEST/sharibako"
echo "Installed sharibako to $DEST/sharibako"
"$DEST/sharibako" --version
```

Executable bit (`chmod +x`).

- Do not test with real Keychain / real Touch ID. Every test uses `FileAgeKeyProvider`.

## Stop Condition

**If the ArgumentParser subcommand model doesn't cleanly support the `key` parent → `generate/import/export` child pattern** (some ArgumentParser versions handle nested subcommands with quirks), stop and surface the actual shape you can get. The user-visible CLI needs `sharibako key generate`, not `sharibako key-generate`. If the framework fights this, we may need a naming compromise — but ask before making it.

**If `KeychainAgeKeyProvider` requires new entitlements or code-signing** that the local `swift build` binary doesn't have, stop and surface. The overview assumes we can hit Keychain from an un-notarized dev binary during dogfooding; if macOS refuses without entitlements, that's a real design constraint that affects both this AT and Phase 6 distribution planning.

**If `AsyncParsableCommand` and `swift-testing` interact badly with the in-process `runCommand` helper** (async main function, harness quirks), stop and pick between: (a) invoke commands directly via their `.run()` methods with hand-parsed args; (b) shell out to the built binary for command tests; (c) something else. Don't quietly ship subprocess-only testing for what should be in-process.

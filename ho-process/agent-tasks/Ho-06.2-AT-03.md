# Ho-06.2-AT-03 — Settings scene + scan-root footer

**Goal**

Add a native Settings scene (⌘,) with a System/Light/Dark appearance override
that persists across relaunch, and complete the sidebar footer's waymarking by
naming every configured scan root (full paths in the tooltip). Independent of the
glyph and heal work in AT-01/AT-02.

**Context**

ho-06.2 Decisions 4 and 5 (`ho-process/hos/ho-06.2-glyphs-and-heal.md`), both
gate-born. The operator asked for an appearance flipper; a native `Settings {}`
scene is its home and wires ⌘, automatically. Appearance is a pure UI preference
→ `@AppStorage` (UserDefaults) driving `.preferredColorScheme` — not `config.yaml`
(which holds operational vault/scan state).

The footer already names the vault and remote (ho-06.1 AT-02, in
`ScopeSidebar` + `WorkshopModel+Waymarking.swift`). Scan roots are the missing
half of "where am I / where is it scanning." `WorkshopModel.scanRoots` is already
`[URL]`, so the footer renders *all* configured roots; this task adds read-only
visibility, not management (multi-root add/remove is an owed near-term ho, not
this one).

New scenes must spell `id:` where they take one — but `Settings {}` is its own
scene type keyed by the system, not an `openWindow(id:)` window, so the ho-06.1
positional-id gate bug does not apply here. Do not add a `WindowGroup`.

**Files**

- Modify: `Sources/Sharibako/App.swift` — add the `Settings {}` scene to the
  `App` body; apply the stored appearance to the main window content via
  `.preferredColorScheme`.
- Create: `Sources/Sharibako/Views/SettingsScene.swift` — the settings form and
  the appearance enum + `@AppStorage` binding.
- Modify: `Sources/Sharibako/WorkshopModel+Waymarking.swift` — scan-root
  short/full description accessors for the footer (read-only, over `scanRoots`).
- Modify: `Sources/Sharibako/Views/ScopeSidebar.swift` — the footer's scan-root
  line.
- Modify: `Tests/SharibakoTests/` — unit tests for the appearance enum ↔
  `ColorScheme?` mapping and the scan-root footer description accessors.
- Modify: `.github/workflows/ci.yml` — exclude `SettingsScene.swift` (and any
  new View file) by name with a justification comment.

**Required Changes**

1. **Appearance preference.** A small enum (e.g. `AppAppearance: String,
   CaseIterable` with `system`, `light`, `dark`), stored via
   `@AppStorage("appearance")`. A pure, tested mapping from the enum to
   `ColorScheme?` (`system → nil`, `light → .light`, `dark → .dark`). Keep the
   mapping as a static function or computed property so it is unit-testable
   without a running scene.

2. **Settings scene** in `App.swift` + `SettingsScene.swift`. Add `Settings {
   SettingsScene() }` to the `App` body. `SettingsScene` renders a native
   `Form`/`Picker` for the three appearance options. Apply the resolved
   `ColorScheme?` to the main `WorkshopWindow` content via
   `.preferredColorScheme(...)` reading the same `@AppStorage` value, so a change
   in Settings updates the main window live and survives relaunch.

3. **Scan-root footer accessors** in `WorkshopModel+Waymarking.swift`, read-only
   over `scanRoots`:

   - A short description for the footer: the configured roots abbreviated for
     `home` (reuse the existing `abbreviate(_:against:)` helper's approach —
     against the model's injected `home`, never `NSHomeDirectory()`), joined for
     display; a plain "No scan root configured" when `scanRoots` is empty.
   - A full description for the tooltip: the full paths of all roots.

   Unit-test both against injected `scanRoots` (empty, one root, multiple roots)
   with an injected `home`.

4. **Footer scan-root line** in `ScopeSidebar`, beneath the vault + remote lines,
   same visual treatment: short form inline, full paths in the help tooltip. Not
   shown in the `.noVault` state (consistent with the rest of the footer).

# Ho-06.1-AT-02 — Waymarking + status announce

**Goal**

Make the Workshop name where it is and announce what it does: a sidebar footer
showing the vault path and git remote (via a new `Conduit.remoteURL()`), the
selected scope's marker target in the detail pane, a jump-to-directory toolbar
button, a green/red pulse on the status surface, a sidebar-aligned status
line, always-visible toolbar button titles, an honest Rescan icon, and a
status announce for every creation.

**Context**

All four items answer ho-05 dogfood-gate findings: the operator could not tell
which repo the Workshop was on, where a scope's secrets would land, or whether
buttons had done anything (hover tooltips failed in practice; silent success
read as broken). Decisions 3, 4, and 6-announce in
`ho-process/hos/ho-06.1-workshop-responsiveness.md`. Depends on AT-01: the
jump button and marker-target display read the scan cache
(`cachedMarker(forScope:)`); the creation announces ride the pulse.

**Files**

- Modify: `Sources/SharibakoCore/Conduit.swift` (or `Conduit+Remote.swift`) —
  add `remoteURL()`
- Modify: `Tests/` Conduit test files — `remoteURL()` unit tests
- Modify: `Sources/Sharibako/WorkshopModel.swift` (remote description state,
  creation announces, marker-target lookup for the detail pane)
- Modify: `Sources/Sharibako/Views/ScopeSidebar.swift` (footer)
- Modify: `Sources/Sharibako/Views/SecretDetail.swift` (marker target)
- Modify: `Sources/Sharibako/Views/WorkshopWindow.swift` (jump toolbar block,
  pulse, status alignment, visible labels, Rescan icon)

**Required Changes**

1. **`Conduit.remoteURL() -> String?`** in Core. Shells
   `git remote get-url origin` (the `Conduit.log` pattern: git stays inside
   the Conduit); returns the URL string, or `nil` when no remote is
   configured. Unit tests: a repo with a remote returns it; a repo without
   returns `nil`; a non-git directory follows the existing `Conduit` error
   posture.

2. **Sidebar footer** in `ScopeSidebar`. A fixed block at the bottom: vault
   directory (abbreviated for home, e.g. `~/.sharibako/vault`) and the remote
   in short form; full vault path and full remote URL in the tooltip. "No
   remote" states plainly when `remoteURL()` is `nil`. The model resolves and
   stores the remote description once alongside the launch scan (a fast local
   git call — ride the same launch task, not per-render). No footer in the
   `.noVault` state.

3. **Marker target in the detail pane.** When a scope is selected,
   `SecretDetail` shows the scope's marker directory and materialize target
   from the AT-01 cache, with honest empty states: "Not scanned yet" before
   the cache holds anything, "No marker found for this scope" on a warm-cache
   miss.

4. **Jump-to-directory toolbar button.** Its own `ToolbarItem` block placed
   left of Sync (operator-specified). Opens the selected scope's cached marker
   directory in Finder (`NSWorkspace`). Disabled when no scope is selected or
   the cache holds no marker for it; help text names why. On success, set
   `statusMessage` naming the opened path (every action announces).

5. **Status pulse.** When `statusMessage` changes, the status surface flashes
   a brief green background tint; when `errorMessage` changes, red. The text
   stays (no auto-clear). Respect Reduce Motion (fall back to a non-animated
   emphasis). Implementation shape is the executor's call; keep any branching
   in the model or a small testable helper if it grows.

6. **Status line alignment + toolbar labels + Rescan icon.** Align the status
   line's leading edge with the sidebar column's section labels (not the
   window edge). Toolbar action buttons render icon + always-visible title.
   Replace Rescan's `magnifyingglass` with a symbol that reads as re-scan
   (e.g. a rotate/refresh form — executor picks; not a search glyph).

7. **Creation announces.** `addScope`, `addSecret`, `addSharedEntry` set
   `statusMessage` on success: "Created scope <id>.", "Added <key> to
   <scope>.", "Created shared entry <id>." Unit-test the messages in the model
   tests.

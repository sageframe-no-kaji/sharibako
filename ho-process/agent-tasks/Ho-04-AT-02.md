# Ho-04-AT-02 â€” Write verbs: get, add, rotate, link, unlink, materialize, update, sync, clean

## Goal

Implement the nine remaining ho-04 verbs on top of AT-01's scaffold. Each is a thin adapter over a `SharibakoCore` call: parse args, resolve vault + age key + (optionally) scope from cwd, invoke the underlying operation, render human or JSON output, exit with the right code. Touch ID prompts (via `AgeKeyProvider.loadIdentity`) fire once per command on the decrypt path â€” never per-key. `materialize --force` maps to `overwriteDrift: true`. Scope-arg omission triggers cwd-based marker resolution for the Materializer-triad verbs (`materialize`, `update`, `clean`, `heal` â€” the last of which was already wired in AT-01). Full test coverage against `FileAgeKeyProvider`.

## Required Changes

### 1. `ScopeResolver.swift` (small helper)

```swift
enum ScopeResolver {
    /// If `explicit` is non-nil, returns it as-is.
    /// Otherwise resolves `.sharibako` walking up from `cwd` (or the passed URL),
    /// loads the marker, returns `marker.scope`.
    static func resolve(
        explicit: String?,
        startingFrom: URL,
        materializer: Materializer
    ) throws -> (scopeID: String, marker: ScopeMarker?)
    // marker is nil when scopeID was passed explicitly and no cwd walk happened.
}
```

Verbs that use it: `materialize`, `update`, `clean`. Not `get`, `add`, `rotate`, `link`, `unlink` â€” those require an explicit scope even in a project dir (see Decision 7).

### 2. `ValueInput.swift`

```swift
struct ValueInput {
    let value: String?
    let fromStdin: Bool

    func read() throws -> String {
        // Exactly one of value / fromStdin must be set; otherwise userError.
        // fromStdin reads all of stdin, strips a single trailing \n if present.
    }
}
```

Reused by `add` and `rotate`.

### 3. `GetCommand.swift`

`sharibako get <scope> <key>`.

1. Resolve vault + age key. `AgeKeyProvider.loadIdentity(reason: "Decrypt sharibako secret \(key)")` â€” Touch ID prompt on macOS.
2. `try VaultCore(vaultURL:, ageKeyURL:).getValue(key, inScope: scope)`.
3. Print raw value + trailing newline on stdout.
4. Errors: `secretNotFound` â†’ exit 2 with "Secret `<key>` not found in scope `<scope>`."; other errors via `ErrorReporter`.
5. No `--json` (the value IS the payload).

### 4. `AddCommand.swift`

`sharibako add <scope> <key> [--value <v> | --from-stdin] [--notes <text>]`.

1. `ValueInput.read()` for the value.
2. Resolve vault, verify scope exists via `vaultCore.getScope(scope)`; if missing, exit 2 with "Scope `<scope>` does not exist. Create it first."
3. `vaultCore.addSecret(key, value:, inScope: scope, notes:)` â€” needs age key for encrypt (`ageKeyURL` init). Touch ID prompts once at command start.
4. Print `"Added <scope>/<key>"` on stdout (unless `--json`, then `{"added": {"scope":"...","key":"..."}}`).
5. Refuse to overwrite existing `<KEY>.age` silently: check `vaultCore.inspect(scope)` first; if key already exists, exit 2 with "`<key>` already exists in `<scope>`. Use `sharibako rotate` to change its value." Add `--force` flag to bypass.

### 5. `RotateCommand.swift`

`sharibako rotate <scope> <key> [--value <v> | --from-stdin]`.

1. `ValueInput.read()`.
2. Look up whether the key is a scope-local `.age` or a `.link` via `vaultCore.inspect(scope)`. If `.link(sharedID)`, call `vaultCore.rotateShared(sharedID, newValue:)`. If `.value`, call `vaultCore.rotate(key, inScope: scope, newValue:)`. If neither exists, exit 2 with "Secret `<key>` not found in scope `<scope>`."
3. Print `"Rotated <scope>/<key>"` or `"Rotated shared entry <sharedID> (via <scope>/<key>)"`.
4. Touch ID once (rotate decrypts existing to preserve notes, then re-encrypts).

### 6. `LinkCommand.swift`

`sharibako link <scope> <key> <shared-id>`.

1. Verify `shared-id` exists in `vaultCore.listShared()`. If not, exit 2 with "Shared entry `<id>` does not exist."
2. `vaultCore.link(key, inScope: scope, toShared: sharedID)`.
3. No age key needed (`.link` files are plaintext scope-ID pointers).
4. Print `"Linked <scope>/<key> â†’ shared/<id>"`.

### 7. `UnlinkCommand.swift`

`sharibako unlink <scope> <key>`.

1. Requires age key (decrypt shared, re-encrypt as scope-local). Touch ID once.
2. `vaultCore.unlink(key, inScope: scope)`.
3. Print `"Unlinked <scope>/<key> (now scope-local)"`.

### 8. `MaterializeCommand.swift`

`sharibako materialize [<scope>] [--force]`.

1. `ScopeResolver.resolve(explicit: scopeArg, startingFrom: cwd, materializer:)`.
2. If the resolver returned a marker (cwd-walk path), use it directly. Otherwise `mat.resolveMarker(forScope: scope, scanRoots: [defaultScanRoot])`.
3. `mat.materialize(marker:, overwriteDrift: force)`.
4. Render the result:
   - `.wrote(path, keys)` â†’ `"Wrote <path> (<N> owned key(s): a, b, c)"`
   - `.unchanged(path)` â†’ `"<path> already up to date"`
   - `.diffPending(diff)` â†’ print the drift + tell user to rerun with `--force`; exit code 2 (drift requires user acknowledgment).
5. No `--json` on this verb (write-side; success/exit code carries).

### 9. `UpdateCommand.swift`

`sharibako update [<scope>]`.

1. Scope resolve (same shape as materialize).
2. `mat.update(scopeID:, marker:)`.
3. Render:
   - `.updated(keys, warnings)` â†’ "Updated `<N>` key(s): a, b, c" + optional warnings section
   - `.noChanges(warnings)` â†’ "No changes" + optional warnings
   - `.fileMissing(path)` â†’ exit 2 "No `.env` at `<path>` to read"
4. Warnings print to stderr in yellow (`OutputRenderer.warn`).

### 10. `SyncCommand.swift`

`sharibako sync [--no-push] [--no-pull] [--message <msg>]`.

1. Open `Conduit(vaultURL:)`.
2. `conduit.commit(message: message ?? "sharibako auto-commit")` â€” treat `.nothingToCommit` as fine (print "nothing to commit" and continue).
3. Unless `--no-push`: `conduit.push()`. On `.rejected(reason)`, print reason + suggest `sharibako sync` again after resolving remotely. Exit 5.
4. Unless `--no-pull`: `conduit.pull()`. On `.abortedConflict(conflicts)`, print each conflict file + suggest manual resolution. Exit 5.
5. Print `"Synced: committed <sha>, pushed <n>, pulled <n>"` on success.

### 11. `CleanCommand.swift`

`sharibako clean [<scope>] [--yes]`.

1. Scope resolve.
2. `mat.clean(marker:)`.
3. Confirm before deletion unless `--yes` â€” print `"About to remove <N> owned key(s) from <path>. Continue? [y/N]"` and read one line from stdin. If not `y`/`yes`, exit 0 without action.
4. Render:
   - `.cleaned(path, keys, fileStillExists)` â†’ "Removed `<N>` key(s) from `<path>`" or "Removed `<N>` key(s); `<path>` was empty and has been deleted"
   - `.fileMissing(path)` â†’ "Nothing to clean at `<path>`"
5. No age key needed (clean doesn't decrypt).

### 12. `RoundTripCommandTests.swift` â€” integration

End-to-end flow, one test:

1. Ephemeral vault + age key fixture, ephemeral project dir.
2. `sharibako key import <fixture path>` (via `FileAgeKeyProvider`).
3. `sharibako add kanyo-dev API_KEY --value sk-live-abc`.
4. `sharibako get kanyo-dev API_KEY` â†’ assert stdout is `sk-live-abc\n`.
5. Write a `.sharibako` marker + empty `.env` in the project dir.
6. `sharibako materialize kanyo-dev` (or better: cd into project dir, run `sharibako materialize` with no scope).
7. Assert `.env` contains `API_KEY=sk-live-abc`.
8. Hand-edit `.env` to change `API_KEY` value.
9. `sharibako update` â€” assert exit 0 + "Updated 1 key(s)".
10. `sharibako get kanyo-dev API_KEY` â€” assert new value.
11. `sharibako clean --yes` â€” assert `.env` gone (single owned key, file becomes empty).

This test exercises the full ho-04 loop end-to-end. If it passes, dogfooding is real.

## Do Not

- Do not implement `sharibako init` (ho-04.2).
- Do not implement `sharibako run` (ho-04.5).
- Do not add a scope-arg-optional variant to `get`, `add`, `rotate`, `link`, or `unlink`. Scope resolution from cwd is only for the Materializer-triad verbs (per Decision 7).
- Do not add `--json` to write verbs (except where explicitly listed above for `update`). Success/failure is via exit code.
- Do not modify `SharibakoCore` unless a genuinely missing API surfaces during implementation â€” if so, stop and surface it (same discipline as AT-02 of ho-03).
- Do not batch Touch ID prompts across commands â€” each `sharibako` invocation gets its own prompt if it needs the private key.

## Stop Condition

**If `sharibako sync`'s error surface for merge conflicts isn't cleanly renderable** (e.g., the `ConflictedFile` structure from ho-02 doesn't carry the info the CLI wants to display), stop and surface the gap. The overview's stated intent is "structured file list, resolution UX deferred to surfaces" â€” this AT IS one of those surfaces. If the underlying types don't support a good CLI presentation, that's a `Conduit` API gap to name.

**If interactive `y/N` prompts** (for `clean` without `--yes` and `key import` without `--delete-source`/`--keep-source`) can't be tested cleanly in the in-process CLI test framework, stop and surface. Options: mock stdin, require flags in tests, or shell out. Don't ship a `y/N` prompt with no test coverage of both branches.

**If `sharibako sync` needs the age key** (it shouldn't â€” git operations are secret-agnostic), stop and surface. This might indicate we're accidentally opening `VaultCore(vaultURL:ageKeyURL:)` in a code path that only needs `VaultCore(vaultURL:)`.

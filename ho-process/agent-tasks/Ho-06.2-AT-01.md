# Ho-06.2-AT-01 — Three-state glyphs + orphan rows

**Goal**

Give every sidebar scope row a glyph that reads its state from ho-06.1's
in-memory scan cache — `live_here` when a marker for the scope sits in the scan
roots, `live_elsewhere` when none does — and render orphaned markers (and
malformed-marker scan failures) as rows in a distinct "Unlinked markers" sidebar
section. All state is computed from cached data already in `WorkshopModel`; no
tree walk, no decryption, no Touch ID.

**Context**

ho-06.2 Decision 2 (`ho-process/hos/ho-06.2-glyphs-and-heal.md`). The scan cache
was built in ho-06.1 exactly so glyphs never trigger a walk:
`WorkshopModel.scanReport` holds a `ScanReport` (`.markers` + `.failures`),
populated at launch and on Rescan. `Materializer.status(scopeID:scanRoots:)`
exists in Core but **re-walks the tree** (it calls `scan(roots:)` internally) —
do not use it; compute from the cache. `scopes` is the loaded vault scope list.

Glyph state maps to a *scope*: a `live_here`/`live_elsewhere` value belongs on a
scope row. An `orphaned` marker references a scope the vault doesn't have, so it
has no scope row — it becomes its own row under "Unlinked markers." This task
surfaces orphans only; remediation (create-scope, remove-marker) is ho-06.3.

This is the first of three tasks. AT-02 (heal) extends the scope row view you
build here with a drift badge — build the row so an additional trailing
indicator has a natural place, but do not build the badge itself.

**Required Changes**

1. **`GlyphState` enum + `glyphState(forScope:)`** in `WorkshopModel` (or the
   `+Glyphs` extension). A two-case enum for scope rows:

   - `.liveHere` — `scopes` contains the scope AND `scanReport?.markers`
     contains a marker whose `scope` equals it.
   - `.liveElsewhere` — `scopes` contains the scope AND no cached marker matches.

   `glyphState(forScope:)` returns the state for a given scope id, reading only
   `scopes` and `scanReport` (both already owned by the model). Synchronous, pure
   over current state. When `scanReport` is nil (no scan yet), every in-vault
   scope reads `.liveElsewhere` — the honest pre-scan answer (we have not found a
   marker here). Unit-test each branch, including the cold-cache case.

2. **Orphan + failure derivation** in the same place. A computed accessor
   returning the "Unlinked markers" contents from the cache:

   - **Orphaned markers:** entries of `scanReport?.markers` whose `scope` is not
     in `scopes` — a marker pointing at a vault scope that doesn't exist. Carry
     the marker's `scope` string and `markerURL`.
   - **Scan failures:** `scanReport?.failures` (malformed markers, ho-04.11) —
     carry `markerURL` and `reason`.

   Model these as a small `Sendable`/`Equatable` value (e.g. an
   `UnlinkedMarker` enum with `.orphaned(scope:markerURL:)` and
   `.failed(markerURL:reason:)` cases) so the view renders without re-deriving.
   Empty when the cache is cold or holds none. Unit-test derivation against a
   scan report containing a valid marker, an orphaned marker, and a failure —
   assert only the orphan and failure surface.

3. **Scope row glyphs** in `ScopeSidebar`. Each scope row shows an SF Symbol
   reflecting `glyphState(forScope:)`, with the distinction carried by shape (not
   color alone — colorblind-safe) and a help tooltip naming the state
   ("Materialized here" / "Lives on another machine"). Leave a trailing slot in
   the row layout where AT-02's drift badge will sit; do not add the badge.

4. **"Unlinked markers" section** in `ScopeSidebar`. A sidebar section, visually
   distinct from the scope-type sections, listing the orphan/failure rows from
   change 2. Each row shows the orphaned scope name (or "malformed marker" for a
   failure) with its `markerURL` path in a help tooltip and the orphaned glyph.
   Selectable to show the path; no action buttons. Omit the whole section when
   there are no unlinked markers. Do not add it to `scopeSections` (that drives
   real vault scopes) — it is its own section below them.

### 2. `scripts/install.sh` — add signing step

Current shape (before this AT):

```bash
#!/usr/bin/env bash
set -euo pipefail
REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
DEST="${SHARIBAKO_INSTALL_DIR:-/usr/local/bin}"
cd "$REPO_ROOT"
echo "Building sharibako (release)..."
swift build -c release --product sharibako
BINARY_PATH="$REPO_ROOT/.build/release/sharibako"
if [[ ! -f "$BINARY_PATH" ]]; then
    echo "Error: build succeeded but binary not found at $BINARY_PATH" >&2
    exit 1
fi
mkdir -p "$DEST"
install -m 0755 "$BINARY_PATH" "$DEST/sharibako"
echo "Installed sharibako to $DEST/sharibako"
"$DEST/sharibako" --version
```

After: insert a signing step between the build check and the `install -m 0755`, gated on `uname -s = Darwin`.

```bash
# Sign on macOS. Non-macOS platforms fall through without signing.
if [[ "$(uname -s)" == "Darwin" ]]; then
    SIGNING_IDENTITY="Developer ID Application: ANDREW TODD MARCUS (3N8F759K8D)"
    ENTITLEMENTS="$REPO_ROOT/sharibako.entitlements"

    # Verify the cert is present before attempting to sign.
    if ! security find-identity -v -p codesigning | grep -q "3N8F759K8D"; then
        echo "Error: signing identity '$SIGNING_IDENTITY' not found in Keychain." >&2
        echo "Install your Developer ID Application cert (Apple Developer portal → Certificates)." >&2
        exit 1
    fi

    echo "Signing sharibako with Developer ID..."
    codesign \
        --sign "$SIGNING_IDENTITY" \
        --entitlements "$ENTITLEMENTS" \
        --options runtime \
        --timestamp \
        --force \
        "$BINARY_PATH"

    echo "Verifying signature..."
    codesign --verify --verbose "$BINARY_PATH"
else
    echo "Signing skipped (not macOS: $(uname -s))."
fi
```

Key details:
- `set -euo pipefail` is already at the top; any codesign failure aborts the whole script.
- `security find-identity -v -p codesigning | grep -q "3N8F759K8D"` — check by team ID (short, unique to the practitioner). Not by full cert string (spaces + parens make grep fragile).
- `codesign --verify --verbose` after signing catches "signature applied but invalid" states early. `spctl --assess` is NOT included — locally-signed unnotarized binaries fail spctl by design, and that's fine for dogfooding.
- `--timestamp` requests Apple's timestamp server signature. Nice for provenance; harmless if the network is unavailable (codesign continues without).
- No `--deep` needed — sharibako is a single binary, not a bundle.

### 3. On non-macOS: graceful skip

The `if [[ "$(uname -s)" == "Darwin" ]]` guard means Linux runs the install script and gets a working `sharibako` binary with signing skipped — Linux use continues to depend on `--age-key <path>` and passphrase-encrypted key files as designed in ho-04 Decision 3.

## Do Not

- Do not modify any Swift code. `KeychainAgeKeyProvider.swift` is correct; it just needs a signing identity.
- Do not add specific entitlements (e.g., `com.apple.developer.keychain-access-groups`) unless `codesign --options runtime` explicitly refuses to sign without them. Empty entitlements is the correct start.
- Do not implement notarization. That's for distribution (Phase 6), not for local dogfooding.
- Do not add automated tests for the signing step. Manual verification is the acceptance check.
- Do not include `spctl --assess` in `install.sh`'s verification. Locally-signed unnotarized binaries fail spctl by design; that's not a bug, and printing that failure would be misleading.
- Do not hardcode the SIGNING_IDENTITY in a way that requires editing when the practitioner's team ID changes. But leaving the exact string as a constant near the top of the script is fine — the practitioner has one team.

## Stop Condition

**If `codesign --options runtime` refuses to sign with an empty entitlements file**, stop and surface the exact error message. There's a small chance macOS requires at least one entitlement for the CLI shape; if so, name what's needed rather than guess.

**If `security find-identity` doesn't detect the Developer ID cert** even though the practitioner has confirmed it's present in their login Keychain, stop and surface. Might be a keychain-access-list issue or a specific search filter problem. Don't just remove the pre-check — the check exists to give a clean error, and if it's spuriously failing we need to know why.

**If `sharibako key generate` on the signed binary STILL returns errSecMissingEntitlement**, stop and surface. That would mean signing alone isn't enough and there really IS an entitlement Apple requires for biometry-gated Keychain access from a CLI. Prior guidance says signing alone is enough; if that's wrong for this specific case, surface with the actual error before adding anything.

**If any of the existing 200+ Swift tests break** after this AT, stop. This AT should be behavior-neutral for the test suite. Any test change is a regression to investigate.

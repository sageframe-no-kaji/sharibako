# Ho-06.7-AT-01 — Core: deleteScope + deleteSharedEntry

**Goal**

Add two filesystem-only vault operations to `SharibakoCore`: `deleteScope(_ id:)`
removes a scope's whole directory, and `deleteSharedEntry(_ id:force:)` removes a
shared entry — refusing when other scopes link it unless forced. Neither decrypts
anything; both are reachable through the keyless `VaultCore.init(vaultURL:)`. Full
filesystem-test coverage, no `age` required.

**Context**

ho-06.7 Decisions 1–3 (`ho-process/hos/ho-06.7-delete-verb.md`). Deletion is the
missing inverse of `createScope` / `addSharedEntry` — the system design says
deletion "always … only ever touches the vault" (kamae-2 line 174), and this task
is the Vault Core operation that sentence assumes. A scope is a directory at
`scopes/<id>/` (`scope.yaml` + `<KEY>.age` + `<KEY>.link` files); a shared entry
is a single `shared/<id>.age` file. `createScope` lives in
`Sources/SharibakoCore/VaultCore.swift` (line ~192) and `linkGraph()` in the same
file (line ~308) — both new verbs belong beside them (no `+Encryption` file: no
crypto). `VaultLayout.scopeDirectoryURL(_:in:)` validates the identifier at the
sole path-building chokepoint; route scope-path building through it so traversal
validation is inherited, not re-implemented.

**Files**

- Modify: `Sources/SharibakoCore/VaultCore.swift` — add `deleteScope(_:)` and
  `deleteSharedEntry(_:force:)` beside `createScope`/`linkGraph`.
- Modify: `Sources/SharibakoCore/VaultError.swift` — add the
  `sharedEntryLinked(id:linkers:)` case and its `errorDescription`.
- Modify: `Sources/SharibakoCore/VaultLayout.swift` — only if a shared-entry path
  helper (`sharedEntryURL(_:in:)`) doesn't already exist and one is warranted for
  the chokepoint; reuse the existing helper if present.
- Modify: `Tests/SharibakoCoreTests/VaultCoreFilesystemTests.swift` (or the
  nearest filesystem-test file) — deletion tests over ephemeral temp vaults, no
  `age`.

**Required Changes**

1. **`deleteScope(_ id: String) throws`** in `VaultCore`.
   - Resolve the scope directory through `VaultLayout.scopeDirectoryURL(id, in:)`
     (inherits identifier validation → `invalidIdentifier` for a bad id).
   - If the directory does not exist, throw `VaultError.scopeNotFound(id:)`.
   - Otherwise `FileManager.removeItem(at:)` the directory tree.
   - Touches nothing under `shared/`, no markers, no git.

2. **`deleteSharedEntry(_ id: String, force: Bool = false) throws`** in `VaultCore`.
   - Resolve `shared/<id>.age` (through the shared-path chokepoint / helper).
   - If it does not exist, throw `VaultError.sharedEntryNotFound(id:)`.
   - Compute linkers via `linkGraph()[id]` (the `[(scopeID, key)]` referencing it).
   - If linkers is non-empty **and** `force == false`, throw
     `VaultError.sharedEntryLinked(id: id, linkers: linkers)` and remove nothing.
   - Otherwise `FileManager.removeItem(at:)` the shared file. When `force == true`
     and linkers existed, the `.link` files are intentionally left dangling — do
     not touch them, do not convert them, do not walk scopes to clean up.

3. **`VaultError.sharedEntryLinked(id: String, linkers: [(scopeID: String, key: String)])`**
   in `VaultError.swift`. Add to the enum and give it an `errorDescription` that
   names the id and lists the `scope/key` pairs (e.g. "shared entry 'openai-personal'
   is linked by kanyo-dev/OPENAI_API_KEY, glassroom/OPENAI_API_KEY"). Match the
   surrounding cases' associated-value and message style. If the enum's associated
   values can't carry a tuple array cleanly under the existing `Equatable`/`Sendable`
   conformances, use a small `Sendable`/`Equatable` `struct Linker { let scopeID; let key }`
   and carry `[Linker]`.

4. **Filesystem tests** in the Core filesystem-test target (no `age`):
   - `deleteScope` removes the directory and only it — assert the scope dir is
     gone, a sibling scope survives, and `shared/` is byte-for-byte untouched.
   - `deleteScope` on an absent id throws `scopeNotFound`.
   - `deleteScope` on an invalid id throws `invalidIdentifier` (chokepoint).
   - `deleteSharedEntry` on an unlinked entry removes it.
   - `deleteSharedEntry` on a **linked** entry with `force: false` throws
     `sharedEntryLinked` naming the linkers, and the file is still present.
   - `deleteSharedEntry` on a linked entry with `force: true` removes it and
     leaves the linker's `.link` file in place (assert the file still exists).
   - `deleteSharedEntry` on an absent id throws `sharedEntryNotFound`.

**Do Not**

- Do not decrypt anything or acquire an age key. Deletion is file removal; it uses
  `init(vaultURL:)`, not the encryption path.
- Do not perform any git operation. `sync` commits the removal (Decision 3).
- Do not touch `.sharibako` markers or materialized `.env` files — deletion only
  touches the vault (Decision 3).
- Do not cascade-unlink or clean up dangling links under `--force`. Orphaning is
  the ratified behavior; the dangling links are surfaced by existing heal/orphan
  machinery (Decision 2).
- Do not add the CLI verb or any GUI code — those are AT-02 and AT-03.

**Stop Condition**

If `linkGraph()`'s return shape or the enum's existing `Equatable`/`Sendable`
conformances make `sharedEntryLinked`'s associated value awkward to model, choose
the small-struct form (change 3) and note it — do not change `linkGraph`'s public
signature or other error cases to accommodate it.

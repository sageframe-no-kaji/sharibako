# Ho-03-AT-02 — Materializer read path: ingest, acceptIngest, update, four-way decision matrix

## Goal

Implement the Materializer read path in `Materializer+Ingest.swift`: `ingest(directory:) → ProposedScope` (parses `.env`, `.env.local`, `.env.example`; classifies keys; suggests exact-match shared entries; suggests a scope ID with collision avoidance), `acceptIngest(_:decisions:)` (routes each of the four `KeyDecision` cases to the correct `VaultCore` write + writes the `.sharibako` marker), `update(scopeID:)` (reads current `.env`, updates vault values for owned keys whose file value differs, ignores non-owned lines). Adds the public types `ProposedScope`, `DetectedKey`, `KeyDecision`, `UpdateResult`, plus the scope-ID suggestion helper. Full unit and integration test coverage including the round-trip integrity test that exercises kamae-2.2's byte-for-byte contract across a full materialize → hand-edit → update → materialize cycle.

### 1. `MaterializerTypes.swift` — add public types

Append to the existing file. All types `public`, `Sendable`, `Equatable`.

```swift
public struct ProposedScope: Sendable, Equatable {
    public let directory: URL
    public let suggestedScopeID: String
    public let detectedKeys: [DetectedKey]
    public let suggestedKeysNeedingValues: [String]   // sorted alphabetically
    public let parseWarnings: [ParseWarning]

    public init(directory: URL, suggestedScopeID: String, detectedKeys: [DetectedKey], suggestedKeysNeedingValues: [String], parseWarnings: [ParseWarning])
}

public struct DetectedKey: Sendable, Equatable {
    public let key: String
    public let value: String
    public let sourceFile: URL
    public let nameMatchedSharedID: String?   // set if VaultCore has shared/<key>.age exactly

    public init(key: String, value: String, sourceFile: URL, nameMatchedSharedID: String?)
}

public enum KeyDecision: Sendable, Equatable {
    case importAsLocal(key: String)
    case linkToShared(key: String, sharedID: String)
    case moveToShared(key: String, newSharedID: String)
    case leaveAlone(key: String)
    case skip(key: String)
}

public enum UpdateResult: Sendable, Equatable {
    case updated(keysUpdated: [String], warnings: [ParseWarning])   // warnings may be empty
    case noChanges(warnings: [ParseWarning])                        // warnings may be empty
    case fileMissing(path: URL)
}
```

### 2. `Materializer+Ingest.swift` — read-path operations

Extension file on `Materializer`. Public methods only; private helpers as needed.

**`ingest`:**

```swift
public func ingest(directory: URL) throws -> ProposedScope
```

Algorithm:

1. Suggest a scope ID:
   - Start with `directory.lastPathComponent`
   - Sanitize: lowercase; replace non-`[a-z0-9-]` with `-`; collapse consecutive `-`; trim leading/trailing `-`
   - Query `vaultCore.listScopes()`. If the sanitized name collides, append `-dev`. If that also collides, append `-dev-2`, `-dev-3`, etc.
2. Parse the three candidate `.env` files in order: `.env`, `.env.local`, `.env.example`.
   - Combine `.env` and `.env.local`: `.env.local` values override `.env` values for the same key.
   - `.env.example` only contributes keys where the key is absent from both `.env` and `.env.local`. Those keys go to `suggestedKeysNeedingValues`, not `detectedKeys`.
   - Files that don't exist are silently skipped (not an error).
3. Load `vaultCore.listSharedEntries()` — the ID list.
4. For each detected key with a value: build a `DetectedKey` with `nameMatchedSharedID: <id>` iff a shared entry exists with `id == key` (exact case-sensitive match).
5. Collect all `ParseWarning`s from the parses.
6. Return `ProposedScope`.

`ingest` writes nothing. Marker not created. Vault not touched.

**`acceptIngest`:**

```swift
public func acceptIngest(_ proposal: ProposedScope, decisions: [KeyDecision], scopeID: String? = nil) throws
```

- `scopeID`: if provided, use as the actual scope ID (allows surface layer to override the suggestion). If nil, use `proposal.suggestedScopeID`.
- Verify each decision's `key` corresponds to an entry in `proposal.detectedKeys` (throw a specific error if not — reuse `VaultError.scopeNotFound` isn't right; add a new case to VaultError: `case ingestKeyMismatch(unknownKey: String)`).
- For each decision:
  - `.importAsLocal(key:)`: look up the DetectedKey's value, call `vaultCore.addSecret(scopeID:, key:, value:)` (or the existing equivalent).
  - `.linkToShared(key:, sharedID:)`: verify shared entry exists; call `vaultCore.link(scopeID:, key:, sharedID:)`.
  - `.moveToShared(key:, newSharedID:)`: look up the DetectedKey's value; call `vaultCore.addSharedEntry(id: newSharedID, value:)` then `vaultCore.link(scopeID:, key:, sharedID: newSharedID)`.
  - `.leaveAlone(key:)`: no-op.
  - `.skip(key:)`: no-op.
- Write the `.sharibako` marker at `proposal.directory / .sharibako` with:
  ```yaml
  scope: <scopeID>
  ```
  (Omit `materialize_to`; the default `./.env` is right for ingest.)
- If the scope doesn't yet exist in the vault (no `<KEY>.age` or `<KEY>.link` files ever created because every decision was `.leaveAlone` or `.skip`), create the empty scope directory with a `scope.yaml` anyway. This is a legitimate case — the user might do a two-pass ingest.

For error handling: if `vaultCore.addSecret` fails partway through, the writes so far are not rolled back. Surface the failure; the surface layer decides whether to retry. This matches the ho-01/02 precedent (best-effort partial writes with clear error surfacing).

**`update`:**

```swift
public func update(scopeID: String, marker: ScopeMarker) throws -> UpdateResult
```

Algorithm (per Decision 6 in the ho document):

1. If `marker.targetURL` doesn't exist → return `.fileMissing(path: marker.targetURL)`.
2. Parse the file. Collect warnings.
3. Load the scope's owned keys and current vault values via `vaultCore`.
4. For each owned key present in the parsed file's `keyValue` lines: compare parsed value against vault value.
5. For each key whose file value differs: call `vaultCore.rotate(target: .scope(scopeID), key:, newValue: <file value>)` (or the equivalent existing rotate method). `rotated_at` gets updated.
6. If no keys were updated → `.noChanges(warnings:)`.
7. Otherwise → `.updated(keysUpdated:, warnings:)` with the list of keys whose vault value was rewritten.

Non-owned lines are ignored entirely. Only owned keys' file values are read; the raw text of other lines is only touched by the parser for byte-for-byte preservation of `EnvLine.keyValue` in-place — but for `update` we never write the file, so preservation is moot for this operation.

Repeated `update` on an unchanged file: returns `.noChanges` (deterministic). No re-encryption cycles.

### 5. `Tests/…/MaterializerIngestTests.swift` — ingest suite

swift-testing `@Suite("Materializer Ingest")`. Cover:

**Scope ID suggestion:**
- Fresh vault + directory `bento` → suggestedScopeID `bento`
- Vault has `bento` + directory `bento` → suggestedScopeID `bento-dev`
- Vault has `bento` and `bento-dev` + directory `bento` → suggestedScopeID `bento-dev-2`
- Directory `My Weird Name!` → suggestedScopeID `my-weird-name` (sanitized)
- Directory `___` (all invalid) → suggestedScopeID `scope` (fallback default)

**Detection and merging:**
- Directory with only `.env` → detectedKeys from `.env`, `suggestedKeysNeedingValues` empty
- Directory with `.env` + `.env.local` → `.env.local` values override for shared keys
- Directory with `.env` + `.env.example` where example has a key `.env` doesn't → key goes to `suggestedKeysNeedingValues`
- Directory with only `.env.example` → all keys in `suggestedKeysNeedingValues`, `detectedKeys` empty
- Directory with no `.env`-family files → both lists empty; `ProposedScope` still valid
- Parse warnings from malformed lines in any of the three files → collected in `proposal.parseWarnings`

**Shared name-matching:**
- Vault has `shared/OPENAI_API_KEY.age` + `.env` has `OPENAI_API_KEY` → `nameMatchedSharedID == "OPENAI_API_KEY"`
- Vault has `shared/openai_api_key.age` + `.env` has `OPENAI_API_KEY` → `nameMatchedSharedID == nil` (case-sensitive)
- Vault has multiple shared entries → each detected key matches its own ID at most (no fuzzy)

**Accept ingest:**
- `importAsLocal` → `<KEY>.age` appears in `vault/scopes/<id>/`; decrypts to the ingested value
- `linkToShared` → `<KEY>.link` appears; content is the shared ID
- `moveToShared` with new ID → `vault/shared/<newID>.age` appears AND `vault/scopes/<id>/<KEY>.link` appears; both decrypt to the ingested value
- `leaveAlone` → nothing written for that key
- `skip` → nothing written for that key
- Marker `.sharibako` written at `directory/.sharibako` with `scope: <id>`
- Mixed decisions in a single acceptIngest call → each is honored independently
- `acceptIngest` with all `.leaveAlone` decisions → scope directory and `scope.yaml` still created (empty scope is legitimate)
- `acceptIngest` with a decision referencing a key not in `detectedKeys` → throws `ingestKeyMismatch`
- `acceptIngest` with `linkToShared` for a nonexistent shared ID → throws `sharedEntryNotFound` (existing case)

### 6. `Tests/…/MaterializerUpdateTests.swift` — update suite

`@Suite("Materializer Update")`. Cover:

- `update` on missing file → `.fileMissing`
- `update` with all owned keys matching vault → `.noChanges` (no warnings)
- `update` with one owned key differing → `.updated(keysUpdated: [that_key], warnings: [])`
- `update` with one owned key removed from file → NOT reported (removal is not an update; heal handles that classification)
- `update` with a non-owned line changed → ignored, returns `.noChanges`
- `update` with malformed lines in the file → warnings surface in the result
- `update` twice in a row on the same file → second call returns `.noChanges` (idempotent-observable)
- `update` that succeeds → vault `<KEY>.age` decrypts to the new value; `rotated_at` was updated
- `update` when a `.link` file owns the key → updates the shared entry, not the scope's local copy (rotation semantics apply as ho-01 defines them for shared entries)

### 7. `Tests/…/MaterializerRoundTripTests.swift` — the integrity test

`@Suite("Materializer Round Trip")`. The centerpiece test of AT-02, validating kamae-2.2's byte-for-byte contract.

**The scenario:**

1. Create an ephemeral vault + ephemeral project directory.
2. Write a `.env` with the following exact content (byte-for-byte):
   ```
   # Bento project — dev environment
   #
   OPENAI_API_KEY=sk-fake-openai-value
   DATABASE_URL="postgres://user:pass@localhost/bento_dev"

   # Toggle these as needed
   DEBUG=true
   PORT=3000
   NODE_ENV=development
   ```
3. Run `ingest` → verify `detectedKeys` has all five, in order.
4. Call `acceptIngest` with:
   - `.moveToShared(key: "OPENAI_API_KEY", newSharedID: "openai-personal")`
   - `.importAsLocal(key: "DATABASE_URL")`
   - `.leaveAlone(key: "DEBUG")`
   - `.leaveAlone(key: "PORT")`
   - `.leaveAlone(key: "NODE_ENV")`
5. Assert: marker written; `vault/shared/openai-personal.age` exists; `vault/scopes/<id>/OPENAI_API_KEY.link` exists; `vault/scopes/<id>/DATABASE_URL.age` exists; no `.age`/`.link` for the three left-alone keys.
6. Run `materialize` with `overwriteDrift: false`. The vault values match the ingested values exactly, so this should return `.wrote(path:, keysWritten: [DATABASE_URL, OPENAI_API_KEY])` OR `.unchanged` depending on how the canonical rewrite compares byte-for-byte with the original quoting. **Verify explicitly:** the resulting `.env` on disk has:
   - Preserved the header comment (`# Bento project — dev environment` + `#`)
   - Preserved the blank line between the API key block and the toggles block
   - Preserved `DEBUG=true`, `PORT=3000`, `NODE_ENV=development` byte-for-byte (including no quoting)
   - Preserved the `# Toggle these as needed` comment
   - `OPENAI_API_KEY` line still reads the ingested value (via re-parsing the file)
   - `DATABASE_URL` line still reads the ingested value
7. Hand-edit the file (simulating user edits in an editor):
   - Change `DEBUG=true` to `DEBUG=false` (non-owned edit)
   - Change `DATABASE_URL="postgres://..."` to `DATABASE_URL="postgres://user:newpass@localhost/bento_dev"` (owned edit)
   - Add a new line at the end: `EXTRA=user_added` (non-owned addition)
8. Run `update` for the scope.
   - Assert: `.updated(keysUpdated: ["DATABASE_URL"], warnings: [])`
   - Assert: `vault/scopes/<id>/DATABASE_URL.age` decrypts to the new password value
   - Assert: the vault does NOT contain `DEBUG`, `PORT`, `NODE_ENV`, or `EXTRA` (non-owned; ignored)
9. Now change the vault-side: rotate `shared/openai-personal` to a new value. Simulate this via `vaultCore.rotate(target: .shared("openai-personal"), key: nil, newValue: "sk-fake-openai-rotated")` (or whatever the existing shared-rotate API looks like).
10. Run `materialize` with `overwriteDrift: false`. Should return `.diffPending(diff:)` because `OPENAI_API_KEY` in the file no longer matches the rotated vault value.
11. Assert `diff.ownedKeysDiffering == ["OPENAI_API_KEY"]`.
12. Run `materialize` with `overwriteDrift: true`. Should return `.wrote(path:, keysWritten: [OPENAI_API_KEY])` (or including DATABASE_URL if it re-emits canonical; both are acceptable — verify the file's `OPENAI_API_KEY` now reads the rotated value).
13. Re-read the file. Assert non-owned lines are still exactly as hand-edited in step 7:
    - `DEBUG=false`
    - `PORT=3000`
    - `NODE_ENV=development`
    - `EXTRA=user_added`
14. Assert the header comment and blank-line structure survives.

This one test validates almost every claim of kamae-2.2 end-to-end. If it passes, the ownership model is implemented correctly.

## Do Not

- Do not implement any CLI subcommands. `sharibako update`, `sharibako ingest`, `sharibako materialize` — ho-04.
- Do not add fuzzy matching, case-insensitive matching, or approximate name-matching for shared entries. Exact only.
- Do not persist `.leaveAlone` decisions to `scope.yaml`. Ephemeral in v1 per Decision 8.
- Do not touch non-owned lines in `update` for any purpose beyond what the parser needs to preserve them in memory. Do not read their values into any comparison or diff.
- Do not commit anything via `Conduit`. Materializer never talks to git. Commits happen at the surface layer.
- Do not add new public API beyond §1. Everything else is `private` or `internal`.

## Stop Condition

**If the round-trip test's step 6 result is ambiguous** (e.g., canonical rewrite of the ingested values produces a byte-different file even though semantic values match), stop and surface which lines differ. The kamae-2.2 contract is that ingested-then-materialized values round-trip; if canonical rewrite makes the file "diff clean" but not "byte identical" in a way that surprises users, that's a Decision-5 refinement — not a silent behavior.

**If `VaultCore.rotate` for shared entries requires a different signature** than the scope rotate case (e.g., it needs a separate method or a scope-nil parameter), stop and surface the actual API. `update` needs to call rotate for both scope-local and linked-shared owned keys; the routing needs to be right.

**If the sanitized scope ID sanitizer produces empty output for a directory name** (e.g., the directory is called `___` or `!!!`), stop and surface. The test expects a `scope` fallback default; a real user hitting an all-invalid directory name should see a sensible suggestion, not an empty string.

**If `acceptIngest` cannot atomically write the scope directory + marker** (partial-write halfway means the vault has a scope but no marker, or the marker exists but the scope directory is empty), stop and surface. The kamae-1 seed's "vibe coder unassisted" success criterion requires that init failures leave the filesystem in an inspectable, recoverable state.

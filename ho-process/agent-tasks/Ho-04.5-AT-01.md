# Ho-04.5-AT-01 — `sharibako run` (runtime injection)

## Goal

Build the `sharibako run` verb: decrypt a scope's secrets into memory, spawn a child command with those values set in its environment, inherit stdio, forward signals, and exit with the child's status. Nothing is written to disk. This is the peer output verb to `materialize` (kamae-2.1). Also add the one Vault Core helper `run` needs and register the command.

```sh
sharibako run [--scope <id>] [--dry-run] [--] <command> [args...]
```

**Build rough, dogfood, then harden — in that order.** Get `sharibako run -- sh -c 'echo $FOO'` injecting one secret and propagating exit codes working first, confirm the feel against a real shell, and only then build out the signal-forwarding and exit-mapping test matrix. This is the ho-04.6 discipline (validate rough before hardening) applied to a non-interactive verb: the "feel" to confirm is that a real child process sees the env and that Ctrl-C behaves.

## Required Changes

### 1. `VaultCore.secrets(inScope:)` (Decision 2)

Add to the encryption extension, beside `getValue`:

```swift
/// Decrypts every owned secret in a scope, resolving links, into a key→value dict.
///
/// Loops `inspect(scopeID)` and calls `getValue` on each key — `.link` files
/// resolve to their shared targets exactly as `getValue` already does. Returns an
/// empty dict for a scope with no owned keys. One decrypt per key; the caller
/// unlocks the age identity once for the whole call.
public func secrets(inScope scopeID: String) throws -> [String: String]
```

- Throws `scopeNotFound` when the scope directory is absent (via `inspect`).
- Propagates `linkTargetMissing`, `ageInvocationFailed`, `yamlDecodeError` from `getValue` unchanged.
- No new crypto path — this is the loop `Materializer.loadVaultValues` already does privately, graduated to public.

### 2. `RunCommand.swift` (Decisions 1, 3, 4, 5, 6, 8, 9)

An `AsyncParsableCommand` mirroring `MaterializeCommand`'s skeleton.

Flags/arguments (Decision 8):

```swift
@OptionGroup var global: GlobalOptions
@Option(name: .long, help: "Scope to run under (resolved from cwd when omitted).")
var scope: String?
@Flag(name: .long, help: "Print the names of secrets that would be injected, then exit. No values, no Touch ID.")
var dryRun: Bool = false
@Argument(parsing: .captureForPassthrough, help: "The command and its arguments, after `--`.")
var command: [String] = []
```

Behavior:

- **`--dry-run` (Decision 4).** Resolve the scope (via `ScopeResolver` / `--scope`), open the vault with `VaultCore(vaultURL:)` — **no age key, no provider, no Touch ID** — call `inspect`, print the owned key names one per line, sorted, and exit 0. Never decrypt. (`--dry-run` with an empty command is fine — there is nothing to run anyway.)
- **Empty command with no `--dry-run` (Decision 9).** Throw a usage error (`runCommandEmpty` or ArgumentParser's validation) — there is nothing to spawn.
- **Real run.** Resolve scope → `VaultLocator.resolveProvider` → `provider.loadIdentity(reason: "Decrypt secrets for run")` → `defer handle.release()` → `VaultCore(vaultURL:ageKeyURL: handle.url)` → `secrets(inScope:)`. Exactly `MaterializeCommand`'s unlock sequence.
- **Env merge (Decision 3).** Start from `ProcessInfo.processInfo.environment`, merge the scope secrets on top (`uniquingKeysWith: { _, scopeValue in scopeValue }` — scope wins).
- **Empty scope (Decision 9).** If `secrets` is empty, print a one-line stderr note ("no secrets to inject for scope '<id>'") and run the child with the parent environment unchanged. Not an error.
- **Spawn (Decision 1).** `Foundation.Process`: `executableURL` resolved from the command's first element (search PATH — use `/usr/bin/env <cmd> <args...>` or resolve via PATH lookup so bare command names like `npm` work), `arguments` = the rest, `.environment` = the merged dict, standard handles left at defaults so stdio is inherited. `try process.run()`.
- **Signals (Decision 6).** Install `DispatchSourceSignal` handlers for SIGINT, SIGTERM, SIGHUP (set each to `SIG_IGN` disposition first so the default action doesn't fire, then observe via the dispatch source). On receipt: `kill(process.processIdentifier, sig)`, start a 5-second grace timer, then `kill(pid, SIGKILL)` if `process.isRunning`. Do NOT create a new process group / session for the child (Decision 6 nuance — that breaks the terminal Ctrl-C path). Tear the sources down after the child exits.
- **Wait + exit (Decision 5).** `process.waitUntilExit()`. Then: `.exit` → `Foundation.exit(process.terminationStatus)`; `.uncaughtSignal` → `Foundation.exit(128 + process.terminationStatus)`. Use `Foundation.exit` so the child's status reaches the parent shell rather than being remapped by ArgumentParser.
- **Errors.** Route `VaultError`/`CLIError` through the existing `ErrorReporter.report(error, json: global.json)` as the other commands do. A spawn failure (command not found) is a clean error, not a crash.

Keep the raw signal/`kill`/`DispatchSourceSignal` plumbing behind a small injectable seam where practical so the state logic is testable without actually raising signals at the test process (mirror how other commands inject `lineReader`/factories). The parts that genuinely need a live process/tty are coverage-excluded with a comment naming why.

### 4. Tests

`secrets(inScope:)` (SharibakoCore, ephemeral vault + ephemeral age key):
- Returns every owned key's decrypted value; a linked key resolves to its shared target's value.
- Empty scope → empty dict.
- `scopeNotFound` for an absent scope; `linkTargetMissing` when a `.link` points nowhere.

`RunCommandTests` (SharibakoCLI, `--age-key` to bypass Keychain):
- **Injection end-to-end:** `run -- sh -c 'printf "%s" "$FOO" > out.txt'` (or equivalent through the command's `_run` seam) writes the vault's `FOO` value — assert the child saw it.
- **Env merge / scope wins:** a key present in both parent env and scope resolves to the scope value in the child.
- **Exit-code propagation:** child `exit 3` → wrapper exits 3; child killed by a signal → wrapper exits `128 + signum`. (Drive through the termination-mapping seam if raising a real signal in-test is unreliable.)
- **`--dry-run`:** prints sorted key names, no values, exit 0, and — asserted — never opens an age identity (use a provider/vault path that would fail if decryption were attempted, or assert no Touch-ID/provider call through the seam).
- **Empty scope:** child still runs; stderr note emitted; parent env passed through.
- **Empty command without `--dry-run`:** usage error.
- **Scope resolution:** `--scope` explicit vs. cwd marker walk-up (mirror `MaterializeCommandTests`).

## Do Not

- Do not write any secret value to disk. `run`'s whole reason for existing is the no-file path (kamae-2.1 Class 4 coverage). The only file `run` touches is the temp age-key file the provider already manages and scrubs.
- Do not build `memset_s` scrubbing over the decrypted `[String: String]` (Decision 7 — v1 is minimal best-effort; the key-file scrub via `handle.release()` is the whole story).
- Do not create a new process group or session for the child (Decision 6 — it breaks the terminal Ctrl-C path, which is the common case).
- Do not add a package dependency for process spawning or signal handling — `Foundation.Process` + `DispatchSourceSignal` + `kill(2)` are sufficient.
- Do not build `clean --all` — deferred (parent ho; overview default is explicit-scope-only).
- Do not add `--json` to `--dry-run` — deferred until a real user needs it (overview).
- Do not modify `Materializer`, `Conduit`, or the ingest surface. The only `SharibakoCore` change is the additive `secrets(inScope:)` method.
- Do not rewrite SECURITY.md — reconcile only, and surface drift rather than editing (see Reconcile).

## Stop Condition

- **If `Foundation.Process` does not propagate the child's exit code or signal-death faithfully** (e.g. `terminationReason`/`terminationStatus` don't distinguish clean exit from signal death the way Decision 5 needs), stop and surface — the fallback is `posix_spawn` + `waitpid`, which is a materially different implementation and wants the practitioner's sign-off before you build it.
- **If SIGINT forwarding cannot be made to leave the terminal in a clean state** after the child dies (orphaned child, wrapper hangs on `waitUntilExit`, double-delivery causes a crash rather than a harmless no-op), stop and surface — signal correctness is the part of `run` most likely to bite a real dev server, and a wrong default is worse than an explicit open question.

# Ho-04.7-AT-01 — `sharibako run` feedback (status line + signal-shutdown countdown)

## Goal

Give `sharibako run` two stderr-only feedback surfaces — a startup status line and a signal-shutdown countdown — behind a TTY/flag gate, without changing any of `run`'s behavior. The feedback logic (formatters + gate) is unit-tested through an injected sink; the live signal plumbing stays coverage-excluded.

## Required Changes

### 1. `RunFeedback.swift` — sink + formatters + gate

- **A `Sendable` feedback sink.** A small value type wrapping a `@Sendable (String) -> Void` emitter, with an `emit(_ line:)` method that appends a newline and calls the closure. Provide two constructions: a production one that writes to stderr (`FileHandle.standardError`), and a `disabled` one whose `emit` is a no-op. Sendable so it can cross into `SignalForwarder`'s dispatch handlers.
- **The gate.** A factory (e.g. `RunFeedback.make(json:verbose:isTTY:)`) that returns the stderr sink when feedback should show, else the disabled sink. Rule: show when `isTTY` is true; **suppress** when `json` is true; **force on** when `verbose` is true (overrides a non-TTY). Production passes `isTTY: isatty(STDERR_FILENO) != 0`; tests pass an explicit `isTTY`.
- **Pure formatters** (free functions or static methods returning `String`, no I/O), so they test without a sink:
  - `startupLine(scope:secretCount:command:)` → one line naming the scope, the count, and the command. Count only — **never** secret names or values. Example shape: `sharibako: scope 'diary' — 12 secret(s) → sh -c '…'`. A zero count reads as no secrets injected (this subsumes the old empty-scope note).
  - `forwardingLine(signalName:)` → e.g. `sharibako: forwarding SIGINT to child…`.
  - `countdownLine(secondsRemaining:)` → e.g. `sharibako: waiting for child to exit… 5`. New line per second (no carriage-return overwrite).
  - `sigkillLine()` → e.g. `sharibako: child unresponsive — sending SIGKILL`.
  - A signal-number → name helper (SIGINT/SIGTERM/SIGHUP) for `forwardingLine`.

### 2. `RunCommand.swift` — emit and thread through

- Build the sink in `_run` from `global.json`, `global.verbose`, and the TTY probe. Keep it injectable: add a defaulted parameter (mirror the existing `forwardSignals` default) so tests pass a capturing sink and a forced-on gate.
- Emit `startupLine(scope:secretCount:command:)` through the sink **after** `secrets` is resolved and **before** `spawnAndWait`.
- **Remove** the standalone `secrets.isEmpty` stderr note — the startup line's zero count replaces it. (Keep the behavior: empty scope still runs the child with the parent env; only the message changes.)
- Pass the sink into `spawnAndWait`, which passes it to `SignalForwarder`.
- No other behavioral change: same scope resolution, unlock, env merge, spawn, exit-code mapping.

### 3. `SignalForwarder.swift` — shutdown feedback

- Take the sink in `init` (default to the disabled sink so existing construction stays valid).
- On signal receipt (in the event handler): emit `forwardingLine` immediately, forward the signal (unchanged), and start a **per-second countdown** (`DispatchSourceTimer`) emitting `countdownLine(secondsRemaining:)` from `grace` down toward 1, alongside the existing grace→SIGKILL `asyncAfter`. On escalation, emit `sigkillLine` before `kill(pid, SIGKILL)`.
- In `teardown()` (which runs after the child exits): cancel the countdown timer so a prompt-exiting child prints no stray ticks. A fast exit should show only `forwardingLine` (if the signal fired at all) and nothing after.
- This file stays coverage-excluded — only the message *strings* it emits are covered, via the formatters tested in `RunFeedbackTests`.

### 4. Tests

`RunFeedbackTests.swift` (new, `@Suite`):
- Each formatter returns the expected string; `startupLine` contains the scope and count and **no** secret value or key name.
- The gate: `isTTY: true` → emits; `isTTY: false` → disabled; `json: true` → disabled even when TTY; `verbose: true` → emits even when `isTTY: false`.
- Countdown sequence: `countdownLine` for 5…1 renders the right integers.

`RunCommandTests.swift` (modify):
- **Startup line:** drive `_run` with an injected capturing sink (gate forced on) and assert the captured lines include the startup line with the right scope and secret count.
- **Empty scope:** update the existing test — child still runs, and the captured startup line reports zero secrets (replacing the old note assertion).
- **stdout stays clean:** assert feedback never lands on stdout (the child's stdout is the only stdout).

## Do Not

- Do not change any of `run`'s behavior — spawn, env merge, exit-code mapping, signal set, 5s grace → SIGKILL, no-new-process-group all stay as ho-04.5 shipped. This task is feedback only.
- Do not write feedback to stdout, ever. stdout belongs to the child; `$(sharibako run …)` and pipes must stay byte-clean.
- Do not animate over the child's output (no spinner/redraw during the run). The only in-run feedback is the fixed startup line before the child takes the terminal; the countdown lives only in the signal grace window.
- Do not put secret names or values in any feedback line — count only.
- Do not use carriage-return overwrite for the countdown — plain integer, one line per second, scroll-safe.
- Do not touch `materialize` / `ingest` / `sync` — their terseness is a separate followup.
- Do not build a `--quiet` flag — the TTY gate plus `--json`/`--verbose` is the whole control surface for v1.

## Stop Condition

- If the per-second countdown cannot be made to stop cleanly when the child exits mid-countdown (stray ticks print after the prompt returns, or `teardown()` races the timer into a crash), stop and surface — a countdown that litters the terminal after exit is worse than the silence it replaces, and the fix may want the practitioner's eye on the teardown ordering.

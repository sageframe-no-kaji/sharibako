# Ho-04.4-AT-03 — Dashboard dogfood corrections: viewport, cycle-without-trap, legibility

## Goal

Fix four failures the manual dogfood surfaced in the `IngestDashboard` (AT-01), so the one-page dashboard is actually usable on a real terminal:

1. **Viewport.** The dashboard draws every row on the alternate screen with no windowing, so on a terminal shorter than the content the overflow rows are unreachable (the alternate screen has no scrollback). Add a scrolling viewport that keeps the active row visible at any terminal height.
2. **Cycling `←/→` onto `move` or `link` immediately opens a modal input and traps the user.** Decouple choice-cycling from secondary-input collection: `←/→` only changes the label; the move slug and link target are collected at **commit** time, in cleanly cancelable prompts. Nothing traps the user mid-navigation.
3. **The key column is a monotone wall** ("words words words"). Give the key names their own color, distinct from the choice column.
4. **Footer wording is unclear** ("commit" is jargon; `q` should read "quit").

This is a correction inside the still-open ho-04.4, driven by its dogfood. It is all inside `IngestDashboard` and its tests — `DashboardIngestPrompt`, the `IngestDecisionSource` protocol, `InitCommand`, and `SharibakoCore` do not change. `IngestDashboard.run()` keeps returning `[ResolvedChoice]` in row order.

## Required Changes

All in `Sources/SharibakoCLI/Support/IngestDashboard.swift`.

### 1. Viewport (scrolling window)

- Inject terminal height so the windowing logic is testable without a TTY:
  ```swift
  /// Terminal row count. Defaults to a TIOCGWINSZ query (coverage-excluded);
  /// inject a fixed height in tests.
  var terminalRows: () -> Int = { /* ioctl(TIOCGWINSZ); fallback 24 */ }
  ```
  The default `terminalRows` body (the `ioctl`) is coverage-excluded with a comment, same discipline as the raw-mode reader. The windowing math must be a pure function driven by an injected height and fully tested.
- Compute the visible row window from `terminalRows()`, the banner line count, the footer/picker line count, and the cursor position. Scroll so the active row is always visible. When rows are clipped above or below the window, render a marker line (e.g. `  ↑ 3 more` / `  ↓ 5 more`) so the user knows there's more.
- The window follows the cursor: moving `↓` past the bottom of the window scrolls the window down; `↑` past the top scrolls it up. Keys, marks, and choices for off-window rows are unchanged — only what's drawn changes.
- Factor the window computation as a pure helper (inputs: total rows, cursor index, available height; output: visible range + above/below counts) and unit-test it directly across heights (content taller than window, shorter than window, cursor at top / middle / bottom).

### 2. Decouple cycling from secondary-input; collect at commit

- **`←/→` cycles the label only.** Cycling onto `move` sets the row's choice to a pending move (no slug yet); cycling onto `link` sets a pending link (no target yet). Neither opens a modal. The user can cycle freely through all labels and off again. Remove the "open `slugField`/`linkPicker` on cycle" behavior from `cycleChoice`.
- **Render pending state clearly.** A move row with no slug shows e.g. `move → (set at commit)`; a link row with no target shows `link → (set at commit)`. Still color + the choice label.
- **Collect targets at commit.** On Enter in the main view, before returning: find rows set to move (need a slug) and rows set to link (need a shared-entry pick). If none, return `[ResolvedChoice]` immediately (current behavior). If some, walk them one at a time through the existing in-dashboard `slugField` / `linkPicker` prompts — link defaults to `matchedSharedID` when present — then return once all are resolved. This reuses the sub-prompt rendering already built; it just runs at commit instead of on cycle.
- **Cancelable, never trapped.** During the at-commit collection, **Esc cancels back to the main dashboard** (choices preserved, so the user can fix a row) — it does not abort the run and does not hang (see change 2b). **Ctrl-C** aborts the whole run (`CLIError.aborted`), as now. There is no state the user cannot back out of with a single key.

#### 2b. Lone Esc must not hang

The current raw mode uses `VMIN=1, VTIME=0`, so after `0x1B` the widget blocks forever waiting for a second byte to tell a lone Esc from an arrow escape — Esc appears dead. Fix it so a lone Esc resolves promptly:

- Set the raw-mode read to `VMIN=0, VTIME=1` (a ~100 ms inter-byte timeout) so the default `readByte` returns `nil` on timeout instead of blocking.
- In escape handling, if the byte after `0x1B` is unavailable (timeout → `nil`), treat it as a lone Esc; if it is `0x5B`, parse the arrow. This removes the hang in the main view and in the sub-prompts.
- The main loop already does `guard let byte = readByte() else { continue }`, so a timeout `nil` simply re-polls — keep that. Do not busy-spin tighter than the VTIME period.
- Update the tests that encoded a lone Esc as `[0x1B, 0x00]` (the old peek-consume workaround): a lone Esc is now `[0x1B]` followed by queue exhaustion (injected `readByte` returns `nil`). The injected-byte test path is unaffected otherwise.

If `VMIN=0/VTIME=1` cannot be made to work cleanly within the module-level `termios` pattern without regressing the injected-byte tests, see the Stop Condition.

### 3. Color the key names

- In `renderRow`, wrap the key name in a color distinct from the choice column (which uses green/yellow/cyan/dim). A neutral-but-distinct color (e.g. bold blue or bright white) is fine — pick one and keep it consistent. The active row keeps its `→` marker and bold; the key color still reads underneath the bold.
- Honor no-color contexts if the project has a guard for it; if the dashboard currently emits ANSI unconditionally (it does), match that existing choice — this task is about adding a key color, not reworking color gating.

### 4. Footer wording

Replace the main footer string with clearer language:

```
↑↓ move    ←→ change    space select    a bulk    ⏎ save    q quit
```

(`navigate`→move, `cycle choice`→change, `mark`→select, `commit`→save, `cancel`→quit.) Keep the sub-prompt footers (`Enter confirm  Esc cancel`) but ensure "Esc cancel" is now true (change 2b).

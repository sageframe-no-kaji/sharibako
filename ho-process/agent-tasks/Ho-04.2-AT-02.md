# Ho-04.2-AT-02 — `sharibako init`: the interactive ingest command

## Goal

Implement `sharibako init [<directory>]` on top of AT-01's `InteractiveSelect` and `IngestDecisionSource`. The command drives the full first-run flow: resolve the vault and age recipient, detect an existing marker for reconcile, offer inline key generation when no key exists, run `Materializer.ingest`, confirm the scope ID, walk each detected key through the arrow prompt, and commit the choices via `Materializer.acceptIngest` — writing the vault entries and the `.sharibako` marker. The source `.env` is left untouched. Tests drive the whole flow through a scripted `IngestDecisionSource`; the live terminal is never required in CI.

## Required Changes

### 1. `InitCommand.swift`

`sharibako init [<directory>]` — directory defaults to cwd.

```swift
struct InitCommand: AsyncParsableCommand {
    static let configuration = CommandConfiguration(
        commandName: "init",
        abstract: "Initialize a project directory as a sharibako scope."
    )

    @OptionGroup var global: GlobalOptions

    @Argument(help: "Project directory to initialize (defaults to the current directory).")
    var directory: String?

    @Flag(name: .customLong("no-generate"), help: "Do not offer to generate an age key if none exists.")
    var noGenerate: Bool = false

    func run() async throws {
        do { try _run() } catch { ErrorReporter.report(error, json: global.json) }
    }
}
```

Make an injectable `_run(...)` (mirroring `CleanCommand._run(cwd:lineReader:)`) that accepts: the target directory, the `IngestDecisionSource` (default `InteractiveIngestPrompt`), a line reader for the scope-ID/confirm prompts, and a keygen-decision closure — so tests drive every branch with a `ScriptedIngestDecisionSource` and scripted line input, no terminal.

Flow, in order:

1. **Resolve target directory** (arg or cwd) and the vault URL (`VaultLocator.resolve`).

2. **Re-init detection (Decision 7).** Check `<directory>/.sharibako` directly with `FileManager.fileExists`; if present, `materializer.loadMarker(at:)` it. Announce the bound scope. Offer to reconcile: run `ingest`, then present only detected keys the scope does **not** already own (diff `detectedKeys` against `vaultCore.inspect(existingScope)`), plus surface shared entries now available to link. Never rebind the directory to a different scope without an explicit typed confirmation. Do not use `resolveMarker(startingFrom:)` — direct detection sidesteps the walker's home-boundary bug (see ho-04.2 Decision 7).

3. **First-run key offer (Decision 5).** Determine whether an age key exists: `--age-key` override present → yes; else on macOS check `KeychainAgeKeyProvider().itemExists()`; else check the default Linux key file. If none and `--no-generate` is set, throw a user error instructing `sharibako key generate`. If none and interactive, prompt `No age key found. Generate one now? [Y/n]`; on yes, run the shared generate path (one Touch ID prompt on the signed bundle), print the recipient for backup, continue. On no, exit cleanly.

4. **Ingest.** `let proposal = try materializer.ingest(directory: targetURL)`. If `proposal.detectedKeys` is empty, report "no `.env` secrets found" and still offer to write a marker binding the directory to a scope (an empty scope is legitimate — the practitioner can `add` later); or exit with a clear note. Pick the cleaner behavior and state it in a comment.

5. **Scope ID + type confirm (Decision 3).** Prompt `Scope ID [<proposal.suggestedScopeID>]:` — Enter accepts the suggestion, typed input overrides. Offer scope type (default `.projectDev`) — inline or as a small select. If the chosen ID already exists in `vaultCore.listScopes()`, print a one-line warning ("scope `X` already exists — its keys will be added to") and require confirmation before proceeding (idempotent reuse, not error).

6. **Collect decisions.** `let decisions = try decisionSource.decisions(for: proposal, sharedIDs: try vaultCore.listShared())`. For reconcile (step 2), pass the filtered proposal.

7. **Commit.** `try materializer.acceptIngest(proposal, decisions: decisions, scopeID: resolvedID, scopeType: resolvedType)`. This writes the vault entries and the `.sharibako` marker. No Touch ID fires here (Decision 2 — all writes use the public recipient or write pointers).

8. **Report.** Summarize: scope ID, marker path, and per-decision counts ("imported 2, linked 1, left alone 3, skipped 0"). Do **not** touch `.env` (Decision 4).

### 2. Shared keygen extraction

`init`'s first-run generation must not duplicate `GenerateCommand`'s `age-keygen` shell-out + Keychain/file store + recipient extraction. Extract that into a reusable function (e.g. `AgeKeyBootstrap.generate(destination:global:) -> recipient` in `Support/`) and have both `GenerateCommand` and `InitCommand` call it. Keep `GenerateCommand`'s existing behavior and tests green.

### 3. `SharibakoCommand` registration

Add `InitCommand.self` to the `subcommands` array.

### 4. `InitCommandTests.swift`

Drive `_run` with `CLITestSupport.withEphemeralVaultAndFileKey` (so the `--age-key` file path is used — no Keychain, no biometry in CI) and a `ScriptedIngestDecisionSource`:

- **Fresh init, mixed decisions.** Ephemeral project dir with a `.env` holding several keys; scripted decisions cover all five (`importAsLocal`, `linkToShared` against a pre-seeded shared entry, `moveToShared`, `leaveAlone`, `skip`). Assert: the marker exists and binds the expected scope; imported keys have `<KEY>.age`; the linked key has a `.link`; the moved key created a shared entry + link; left-alone/skip keys wrote nothing; the source `.env` is byte-for-byte unchanged.
- **Scope-ID default vs override.** Enter accepts `suggestedScopeID`; a typed value overrides; a value colliding with an existing scope warns and reuses on confirmation (assert idempotent — existing type preserved).
- **First-run offer.** With no key present and `--no-generate`, `_run` throws the instructive user error. (Keychain generation itself is not exercised headless — cover the branch logic through the injected keygen-decision closure and the file-key path.)
- **Re-init reconcile (Decision 7).** Pre-initialized dir (marker + scope owning key `A`); `.env` now also has key `B`. Assert only `B` is presented to the decision source, the scope is not rebound, and `A` is untouched.
- **Empty `.env`.** No detected keys → the chosen step-4 behavior (assert whichever was implemented).
- **Non-TTY refusal.** With the interactive prompt and a non-TTY input, assert `notInteractiveTerminal` surfaces (drive via the AT-01 seam; the scripted source path is the tested-logic path).

## Do Not

- Do not modify `SharibakoCore` (`Materializer`, `ingest`, `acceptIngest`, `KeyDecision`). Consume as-is; if a genuine gap appears, stop and surface it (same discipline as ho-04's AT-02).
- Do not touch, rewrite, or delete the source `.env` (Decision 4). `init` writes only the vault and the marker.
- Do not use `resolveMarker(startingFrom:)` for re-init detection — direct `fileExists` + `loadMarker` at the target directory (Decision 7), to avoid the walker's home-boundary error.
- Do not build the non-interactive flag surface (`--import-all`/`--yes`/`--scope` as a full path) — deferred (Decision 6). The `directory` arg and `--no-generate` are the only flags this task adds.
- Do not batch or suppress the first-run Touch ID prompt — key generation is one prompt on the decrypt-free public path (Decision 2/5).
- Do not duplicate `age-keygen` logic — extract and share it with `GenerateCommand`.

## Stop Condition

**If reconcile (Decision 7) cannot cleanly present a filtered `ProposedScope`** — e.g. `acceptIngest` rejects a proposal whose `detectedKeys` were narrowed, because it validates decisions against the *original* detected set — stop and surface. The fix might be to filter at the decision layer (pass `.skip` for already-owned keys) rather than reconstruct the proposal, or it might reveal a real `Materializer` API gap.

**If the shared keygen extraction forces a behavior change in `GenerateCommand`** (e.g. its Keychain-vs-file branching doesn't factor without altering the recipient-printing contract), stop and surface rather than silently changing a shipped verb.

**If the empty-`.env` case has no obviously-right behavior** (write an empty-scope marker vs refuse), stop and confirm the intent rather than guessing — this shapes what "initialized" means for a directory with no secrets.

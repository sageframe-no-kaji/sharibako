# Ho-06.2-AT-02 — The heal surface

**Goal**

Make drift between materialized `.env` files and the vault visible and
reconcilable: a Check-drift sweep that decrypts every `live_here` scope behind
one Touch ID through `VaultWorker`, caches a `DriftReport` per scope for the
session, renders drift badges on sidebar rows and per-key drift in the detail
pane from that cache, reconciles a scope via the existing `materialize(force:)`
flow, and reconciles the whole drifted set via a Materialize-all-stale action.
No drift work runs at launch.

**Context**

ho-06.2 Decision 3 (`ho-process/hos/ho-06.2-glyphs-and-heal.md`). The forcing
fact: `Materializer.heal(marker:)` decrypts every owned key (`getValue` in a
loop) to compare vault-plaintext against file-plaintext, so drift cannot be known
without the age key and a Touch ID. That rules out ambient/at-launch drift
badges — they would prompt Touch ID the moment the window opens, undoing
ho-06.1's "window interactive immediately" achievement. So drift is *pulled*,
then *cached for the session*, exactly parallel to how `scanReport` caches
markers.

Two existing surfaces you build on, do not duplicate:

- **`Materializer.heal(marker:) -> DriftReport`** — read-only drift detection.
  `DriftReport.owned` is `[KeyDrift]` with cases `.match`, `.fileMissing`,
  `.fileValueDiffers(vaultSha256:fileSha256:)`, `.fileLineCorrupted`
  (`MaterializerTypes.swift`). Use this for the *view*.
- **`WorkshopModel.materializeSelectedScope(force:)`** — the existing reconcile
  path. It already runs through `VaultWorker`, already acquires the age key,
  already stores a `pendingDiff` and drives a confirmation dialog on drift. Use
  this for the *fix* — do not write a second materialize path.

ho-06.1 Decision 5 gave the shared `LAContext` a 5-minute reuse window, so one
Touch ID covers a multi-scope sweep and a subsequent reconcile within the window.

Depends on AT-01: this task adds the drift badge to the scope row view AT-01
built (the trailing slot it left) and reads AT-01's `glyphState` to know which
scopes are `live_here` (the sweep's candidate set).

**Required Changes**

1. **`driftReports` session cache** on `WorkshopModel`: `private(set) var
   driftReports: [String: DriftReport] = [:]`, keyed by scope id. Never
   persisted (markers and files change externally; a persisted drift cache would
   lie across sessions) — same posture as `scanReport`. Cleared for a scope when
   that scope reconciles.

2. **Check-drift sweep** — an `async` intent in `WorkshopModel+Heal.swift`. Guard
   re-entry against `activity` like every long intent. Acquire the age key on the
   main actor once (user interaction, not CPU work); only its `Sendable` handle
   URL crosses to the worker. Set an `activity` state (add a `.checkingDrift`
   case to `Activity` with a "Checking drift…" label). For each `live_here` scope
   (use AT-01's `glyphState`), resolve its cached marker
   (`cachedMarker(forScope:)`) and run `Materializer.heal(marker:)` through
   `worker`; collect `DriftReport`s into `driftReports`. Release the handle on
   every exit path. Announce the outcome through `statusMessage` (e.g. "Checked
   N scopes — M drifted.") — the sweep visibly concludes even when nothing
   drifted (the silent-success rule). Errors land in `errorMessage`.

3. **Drift classification helper** (tested, pure): given a `DriftReport`, is the
   scope drifted? A scope is drifted when any `owned` entry is
   `.fileValueDiffers`, `.fileMissing`, or `.fileLineCorrupted`; clean when all
   are `.match`. This helper drives both the badge and the Materialize-all-stale
   set. Unit-test it across each `KeyDrift` case, including a report that is all
   `.match` (clean) and one mixing `.match` with `.fileLineCorrupted` (drifted).

4. **Sidebar drift badge** in `ScopeSidebar`, in the trailing slot AT-01 left.
   Renders only when `driftReports` holds an entry for the scope: a shape-distinct
   badge for drifted vs clean, help tooltip naming the drift ("2 keys drifted" /
   "In sync"). No badge before a check has run for that scope — the row shows only
   its AT-01 glyph.

5. **Per-key drift in the detail pane** (`SecretDetail`). When the selected
   scope has a cached `DriftReport`, show per-key drift status from
   `report.owned`: match / file-missing / value-differs / line-corrupted, in
   plain language, never surfacing plaintext (the SHA digests exist only to prove
   difference — do not display them as if meaningful to the user; a "differs"
   label is enough). Honest empty state when no check has run for the scope.

6. **Per-scope reconcile.** Wire a "Reconcile" affordance (in the detail pane or
   the drift badge's context) to the existing `materializeSelectedScope(force:)`
   flow — it already presents the `pendingDiff` confirmation and writes on
   approval. After a successful reconcile, refresh that scope's entry in
   `driftReports` (re-heal it, or clear it and let the next check refill) so the
   badge stops showing stale drift. Do not build a parallel write path.

7. **Materialize-all-stale** — an `async` intent + a toolbar action. Compute the
   drifted set from `driftReports` (change 3's helper). Present a confirmation
   listing the scopes and target paths that will be written. On approval,
   reconcile each through the worker behind one Touch ID (the reuse window covers
   the batch), then refresh their `driftReports`. Announce the batch outcome
   through `statusMessage`. If `driftReports` is empty (no check run yet), the
   action prompts the user to Check drift first rather than silently doing
   nothing.

8. **Chrome fix (Decision 1).** Give Check-drift and Materialize-all-stale
   toolbar homes. If the bar crowds past legibility, move secondary actions
   (Preview .env, Jump-to-directory) into an overflow menu or contextual
   placement — grouping is your call; keep every primary action's title visible
   (the ho-06.1 tooltips-failed lesson). This is native top-toolbar work, not a
   rail.

**Do Not**

- Do not compute drift at launch or on selection change — drift is pulled by an
  explicit Check-drift only. An ambient drift badge means a launch-time Touch ID,
  which is the exact thing Decision 3 designs away.
- Do not write a second materialize path — per-scope and batch reconcile both go
  through the existing `materializeSelectedScope(force:)` and its `pendingDiff`
  confirmation.
- Do not display SHA digests or any plaintext as drift detail — a "differs"
  label is the whole user-facing truth; the hashes exist only to prove
  difference.
- Do not build the right-side tool rail. Decision 1 deferred it; the chrome fix
  here is native top-toolbar plus overflow.
- Do not add rotation-driven "stale" — this ho's Materialize-all-stale is
  drift-driven only; rotation staleness is ho-07.

**Stop Condition**

If the 5-minute Touch ID reuse window does not actually carry the multi-scope
sweep at the dogfood gate (each scope re-prompts), stop and surface — do not
loosen the Keychain query's security posture to force it. This mirrors ho-06.1
AT-03's Stop Condition; the reuse window is a property of the shared `LAContext`,
and if it misbehaves the fix is context lifecycle, not a weaker query.
